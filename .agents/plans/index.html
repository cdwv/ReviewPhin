What to do first

2 · reliability

Post each review safely

If posting stops halfway, a retry must finish the same review without making duplicate or different comments.

3 · queue safety

Group comments that arrive together

Several requests on one merge request or pull request should become one clear review run, without losing a request in a race.

4 · setup security

Make GitHub setup safer

Temporary setup links need safer storage, direct lookup, better permission checks, and clearer repair tools.

5 · repository feature

Load instructions meant only for ReviewPhin

A repository should be able to give ReviewPhin special review rules without giving those rules to every other coding tool.

6 · code health

Load platform data more carefully

Fetch every needed comment, avoid expensive work when it is not needed, and make GitHub and GitLab code easier to follow.

7 · model reliability

Recover malformed model output

Invalid structured output should receive a small number of focused correction attempts before the whole review fails.

8 · tenant safety

Make tenant reassignment explicit

Registering one project through another connection must ask for a decision and always report the assignment actually saved.

9 · local review

Review without writing remotely

Analyze an existing merge request or pull request, make no remote changes, and render the result clearly in the CLI.

10 · model setup

Show available models

List models for the current credentials, check profiles, and fail early when a trustworthy catalog proves a model missing.

11 · authentication docs

Use one clear Copilot token name

Recommend one environment variable, document alias precedence, and separate model authentication from GitHub App access.

12 · GitLab docs

Explain GitLab token permissions

State the Developer minimum, accurate scopes by token kind, and the checks operators should complete before registration.

How each plan is written

1. What is wrong?A short example of the problem a user or operator can see today.
2. What do we want?The result that should be true when the work is finished.
3. How do we fix it?Small steps, checks, and a technical note for the implementing agent.

Every plan talks about the product as it works today and the result we want next. You do not need to know the project’s history to understand the work.
