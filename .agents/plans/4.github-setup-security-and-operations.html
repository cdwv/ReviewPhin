Plan 04 · GitHub setup

Protect setup links and show operators what needs fixing.

GitHub setup uses a secret link for a short time. That secret is stored with the long-lived connection, and ReviewPhin searches every GitHub connection to find it. Setup also needs stronger permission checks and safer tools for large repositories.

ImportanceHigh
Who noticesOperators
Main code areaGitHub App setup
New stored dataSetup session
Finished result: setup links are short-lived and one-use, the raw secret is not stored, broken permissions are explained, repair jobs are controlled, and setup works behind a public URL path such as /reviewphin.

What is wrong today?

The setup secret lives in connection JSON

The raw token is stored with the GitHub connection. A temporary login-like secret should not live beside long-lived app settings.

Finding a setup link means scanning connections

ReviewPhin reads every GitHub connection and opens its JSON until it finds the token. This gets slower as more connections are added.

“Set up” does not always mean “can work”

The GitHub App may be installed but lack permission to create checks, post reviews, or use project memory. Operators need a clear list of missing permissions.

Repair work can be large or deployment-specific

Creating checks for many open pull requests needs progress and limits. Public URLs with a path prefix need a real proxy test, not only string tests.

What do we want instead?

  • Store only a hash of the setup token, like storing a password hash instead of the password.
  • Find a setup session directly by that hash. Do not scan all connections.
  • Reject expired, used, unknown, or wrong-owner setup links.
  • Remove setup sessions when their connection is deleted and clean up expired sessions.
  • Check the GitHub App’s real permissions before calling the connection ready.
  • Show any old GitHub App cleanup work in connection status.
  • Run open-pull-request check creation as an explicit repair job with progress, retry details, and a safe concurrency limit.
  • Make every setup, callback, webhook, docs, asset, and badge route work behind a path prefix.

How do we fix it?

1. Create a one-time linkGive the operator a random token, but store only its hash and expiry.
2. Look it up directlyHash the token from the callback URL and find one setup-session record.
3. Use it onceCheck expiry and owner, mark it used, then continue setup.

New setup-session data

Data Why it exists
Internal ID Names the setup-session record.
Token hash Finds the session without storing the raw secret. It must be unique.
Connection ID Points to the GitHub connection being set up.
Expected owner Stops another GitHub account from finishing the flow.
Created, expiry, and used times Supports expiry, one-time use, cleanup, and troubleshooting.
Small amount of callback state Keeps only information that cannot be read again from the connection. No copied credentials.
Important: putting a token hash back inside connection JSON is not enough. It would still require scanning every connection and would still mix temporary setup state with long-lived connection state.

Work steps

  1. Write the security rules. Cover stolen links, repeated use, expired links, two callbacks at once, wrong owners, deleted connections, and safe logging.
  2. Add the setup-session store. Add matching support to SQLite, Flotiq, and custom-storage documentation.
  3. Change setup routes. Create a raw token only for the URL, hash it before storage and lookup, and safely mark the session used.
  4. Check real GitHub permissions. Tell the operator which permission or repository access is missing. Show pending cleanup from a replaced GitHub App.
  5. Make check creation a controlled repair task. Add progress, limited parallel work, useful retry errors, and safe reruns.
  6. Test a real path-prefix proxy. Put ReviewPhin behind a proxy that removes the prefix and walk through setup, callbacks, webhooks, and static files.
Technical notes for the implementing agent
  • GitHub connection config contains setupToken, setupTokenExpiresAt, and previousAppCleanup.
  • Keep GitHub App credentials, installation identity, connection status, and previousAppCleanup on the platform connection. Only the temporary setup flow moves to the new store.
  • Add direct tokenHash lookup and a safe one-time consume operation. SQLite needs a table and unique index. Flotiq needs the matching content type and mapping.
  • Connection deletion must delete its setup sessions. Tenant deletion does not need special setup-session work.
  • Keep setup HTML in public/github/setup/templates.
  • This work needs a new storage contract version and the full storage verification checklist.

How do we know it works?

  • No raw setup token is stored after the change.
  • A callback finds its session without listing all platform connections.
  • An expired, used, repeated, unknown, or wrong-owner link fails safely.
  • SQLite and Flotiq follow the same lookup, one-use, cleanup, and connection-deletion rules.
  • A connection with missing GitHub permissions is not shown as ready, and the error says exactly what is missing.
  • The repair task can stop and run again without creating duplicate same-commit checks.
  • The full GitHub setup flow works through a real path-prefix proxy.

Checks to run: GitHub setup tests, connection and CLI tests, proxy route tests, storage migration tests, SQLite and Flotiq tests, deletion tests, npm test, and npm run build.
