# Storage-Backed Job Queue Architecture

## Status

Ready to implement.

This is one breaking storage contract revision, `storage-v005`, covering:

- storage-backed interaction-job claiming and leases;
- optional reasoning effort for both model roles;
- reasoning-effort snapshots on interaction runs.

The queue operations must be implemented by each storage adapter. They cannot be
generic `StorageHelpers` operations because SQLite needs adapter-local
transactions to make claims atomic.

## Goals

- Make persisted interaction jobs the queue source of truth.
- Let webhooks, CLI commands, and future trigger sources create storage rows
  without enqueueing process-local work.
- Preserve one active review globally when the storage adapter supports atomic
  claims.
- Recover work after process exit through expiring leases.
- Expire old queued jobs so restored data does not unexpectedly run.
- Preserve retry backoff across process restarts.
- Allow model profiles to configure reasoning effort independently for review
  and text-generation sessions.
- Omit reasoning effort from Copilot SDK session options when it is not
  configured, leaving the harness default unchanged.
- Upgrade the Copilot CLI and SDK so current GPT-5.6 models are available.

### Interaction jobs

Add the terminal status `"expired"` and these fields to
`InteractionJobRecord`:

```ts
type InteractionJobStatus =
  "queued" | "in_progress" | "completed" | "failed" | "cancelled" | "expired";

availableAt: string;
claimToken: string | null;
claimedBy: string | null;
claimExpiresAt: string | null;
latestInteractionRunId: string | null;
```

`availableAt` is initialized to `enqueuedAt`. A retry sets it to the retry time,
so the persisted queue preserves `RETRY_BACKOFF_MS` behavior after restarts.

`claimToken` is a new ULID for every claim attempt. `claimedBy` identifies the
runner process for diagnostics, but ownership checks use `claimToken`. This
fences an old attempt after its lease expires and the job is reclaimed.

Creation, requeue, completion, failure, cancellation, and expiration must set or
clear the new fields consistently:

| Transition           | Required field behavior                                                                                        |
| -------------------- | -------------------------------------------------------------------------------------------------------------- |
| create               | `availableAt = enqueuedAt`; claim fields `null`                                                                |
| claim                | set `status`, `startedAt`, `claimToken`, `claimedBy`, and `claimExpiresAt`; clear `finishedAt` and `lastError` |
| retry                | set `status = "queued"` and future `availableAt`; clear all claim fields and `finishedAt`                      |
| complete/fail/cancel | set terminal status and `finishedAt`; clear all claim fields                                                   |
| expire               | set `status = "expired"` and `finishedAt`; clear all claim fields                                              |

`latestInteractionRunId` starts as `null`. Creating a run for the current claim
sets it to that run id. It remains set through retry and terminal transitions so
observers can select the latest attempt deterministically. Jobs that expire
before starting keep it `null`.

Treat it as a logical id rather than a database foreign key or Flotiq datasource
relation. This avoids a circular job/run deletion dependency; application and
migration tests enforce that non-null values identify runs for the same job.

### Claim-aware store operations

Replace the `InteractionJobStore` alias with an interface that extends the
existing entity-store operations and adds:

```ts
type InteractionJobClaimMode = "atomic" | "single-worker";

readonly claimMode: InteractionJobClaimMode;

claimNext(input: {
  workerId: string;
  claimToken: string;
  now: string;

  jobId: string;
  claimToken: string;
  now: string;
  claimExpiresAt: string;

  jobId: string;
  claimToken: string;

  jobId: string;

  jobId: string;

}): Promise<boolean>;

  jobId: string;
  claimToken: string;
  interactionRunId: string;
  metrics: UpsertInteractionRunMetricsInput;
}): Promise<boolean>;

createCodeReviewSnapshotForClaim(input: {
  jobId: string;
  claimToken: string;
  interactionRunId: string;
  snapshot: CreateCodeReviewSnapshotInput;
}): Promise<CodeReviewSnapshotRecord | null>;

updateReviewFindingStatusForClaim(input: {
  jobId: string;
  claimToken: string;
  interactionRunId: string;
  tenantId: string;
  codeReviewId: number;
  identityKey: string;
  status: ReviewFindingStatus;
  currentStatuses?: readonly ReviewFindingStatus[];
}): Promise<boolean>;

upsertDiscussionMappingForClaim(input: {
  jobId: string;
  claimToken: string;
  mapping: UpsertDiscussionMappingInput;
}): Promise<DiscussionMappingRecord | null>;

reconcileOrphanedInteractionRuns(input: {
  now: string;
  limit: number;
}): Promise<InteractionRunRecord[]>;
```

Claim-aware methods return `false` or `null` when the token no longer owns the
job. Callers treat that as lease loss and must not issue another state update
for that attempt.

`createInteractionRunForClaim` atomically verifies the claim, creates the run,
and sets `latestInteractionRunId`. `transitionInteractionRunForClaim` changes
only an `in_progress` run whose stored attempt token and current job claim both
match. Finding and metric writes use the same ownership predicate and require the
run to remain `in_progress`. This prevents stale attempt data from appearing in
the selected final run.

Snapshot creation, finding-status reconciliation, and discussion-mapping writes
also verify the current claim. The provider request preceding a mapping write
cannot be fenced, but a stale response cannot update shared storage after lease
loss.

`reconcileOrphanedInteractionRuns` conditionally changes only `in_progress`
runs whose job is no longer owned by their attempt token. It returns the runs it
changed so the runner can perform best-effort trigger-lifecycle reconciliation.

`expireQueued` is bounded maintenance separate from claiming. `claimNext`
independently excludes jobs older than `queuedAfter`, so stale jobs cannot run
while expiration catches up across batches.

Generic `EntityStore` mutation methods must not provide a bypass around fencing.
Adapters reject every ordinary mutation of an existing `in_progress` job,
including `upsert`, `upsertMany`, `replace`, `replaceMany`, `update`,
`updateMany`, `patch`, `patchMany`, `delete`, and `deleteMany`. Application
lifecycle changes use claim-aware methods; only queue maintenance inside
`claimNext` may recover a claimed row.

Attempt-scoped snapshots and logs remain associated with their interaction run.
Observers use `latestInteractionRunId`, so data from an abandoned historical
attempt cannot become the reported final result. Provider API calls and shared
discussion reconciliation cannot be made transactional with the storage lease
and remain subject to the exactly-once limitation below.

### Code-review snapshots

Add `interactionRunId: string | null` to `CodeReviewSnapshotRecord`. Existing
snapshots migrate as `null`. Require a non-null run id for new v005 snapshots and
create them through `createCodeReviewSnapshotForClaim`.

When loading prior completed context, pair a run with a snapshot whose
`interactionRunId` matches. Preserve the current latest-snapshot-per-job fallback
only for migrated snapshots whose run id is null.

`claimNext` performs this sequence as one adapter-level claim operation:

1. Recover `in_progress` jobs whose lease expired or whose
   `claimExpiresAt`/`claimToken` is missing. Increment `retryCount`; requeue with
   an immediately eligible `availableAt` when the retry budget remains,
   otherwise fail the job.
2. If any job has `status = "in_progress"` and `claimExpiresAt > now`, return
   no claimed job to preserve global single-review execution.
3. Select the oldest eligible job where:
   - `status = "queued"`;
   - `enqueuedAt >= queuedAfter`;
   - `availableAt <= now`.
4. Order by `availableAt`, then `enqueuedAt`, then `id`.
5. Claim it with the supplied token and return the updated record.

The adapter operation may use private helpers, but lease recovery, active-claim
checking, and selection must not be split across generic read/patch calls for
atomic adapters.

The current Flotiq API has no compare-and-set transaction spanning selection and

Provide a reusable single-worker queue implementation built on the generic
entity store. Flotiq uses it directly, and custom adapters may reuse it when
they declare the same topology. SQLite overrides these operations with its
transactional implementation.

The single-worker helper receives private raw job and run stores so its internal
updates do not go through the public guards against ordinary claimed-row
mutation. Because generic filters have no range operators and Flotiq supports
only one remote order field, the helper exhaustively pages candidate jobs,
filters timestamps in memory, and applies the contract's full deterministic
sort before updating and reading back the claim token.

For `single-worker` adapters:

- one ReviewPhin process may run the job runner;
- additional HTTP processes must have job execution disabled;
- startup logs a warning with this topology requirement.

This limitation must be explicit in deployment and custom-adapter docs. Global
single-review execution across competing processes is guaranteed only by
`"atomic"` adapters. CLI processes may submit jobs and observe storage with
either claim mode because they never claim or execute jobs.

## Runner

Add `src/jobs/storage-backed-job-runner.ts` and remove
`src/jobs/job-queue.ts` after all callers are migrated.

Responsibilities:

- poll the claim-aware interaction-job store;
- create one stable `workerId` per process and one `claimToken` per attempt;
- process no more than one claimed job at a time;
- renew the active claim on a heartbeat;
- abort the active claim context if heartbeat renewal reports lease loss;
- expire old queued jobs in bounded maintenance batches;
- reconcile persisted orphaned runs on every maintenance tick;
- sleep when no eligible job is claimable;
- preserve persisted retry delay;
- expose deterministic methods for server lifecycle and tests.

Public methods:

```ts
start(): void;
stop(): Promise<void>;
```

`start()` and `stop()` are idempotent. `stop()` stops new claims, clears poll
timers, waits for the active attempt to settle while continuing its heartbeat,
then returns so storage can close safely. The first process signal starts this
drain. A second signal logs and terminates the process; the unfinished lease is
left for another runner to recover rather than closing storage beneath live
worker code.

Keep one package-internal `runOnce()`/tick method for deterministic tests rather
than exposing another production lifecycle mode.

The runner passes a claim context containing `jobId`, `claimToken`, an
`AbortSignal`, and `assertOwned()` into `ReviewWorker`. Lease loss aborts the
signal. The worker checks it before each storage commit and provider-side
lifecycle action and after each long-running operation. An in-flight third-party
request may still finish after lease loss, so exactly-once external side effects
remain outside the fencing guarantee.

Run bounded orphan reconciliation on startup and every poll, including polls
where `claimNext` recovers a lease but returns no new job. A run is orphaned when
its claim token is null, its job is no longer `in_progress`, or its token differs
from the job's current token. Mark it failed conditionally from `in_progress`.
This pass is idempotent and state-derived, so another runner can finish it after
a crash without an outbox or ephemeral claim-result metadata.

Trigger-lifecycle reconciliation is best effort and follows this mapping:

- a queued job or an in-progress job with a different token calls `retry`;
- `failed`, `cancelled`, or `expired` calls `failed`;
- `completed` performs no orphan lifecycle call because the winning attempt
  already completed it.

### Configuration

Add validated configuration:

```ini
REVIEWPHIN_JOB_POLL_INTERVAL_MS=2000
REVIEWPHIN_MAX_QUEUED_JOB_AGE_MS=21600000
REVIEWPHIN_JOB_LEASE_MS=120000
REVIEWPHIN_JOB_RUNNER_ENABLED=true
```

Require positive poll/max-age durations and a lease of at least `1000ms`. Derive
the heartbeat interval internally as one third of the lease, rather than adding
another operator setting.

`REVIEWPHIN_JOB_RUNNER_ENABLED=false` supports HTTP-only replicas when a
separate process is the designated runner for a `single-worker` adapter. It
controls automatic runner startup in the server entrypoint. CLI submission and
watching do not depend on this process-local setting.

Expiration uses the original `enqueuedAt`, not `availableAt`, and sets:

```ts
status: "expired";
finishedAt: now;
lastError: "Queued job expired after exceeding the maximum age of <age>ms.";
```

A retried job can therefore expire after earlier processing attempts; expiration
means it exceeded the maximum total job age while queued, not necessarily that
it was never processed.

- oldest eligible job is claimed deterministically;
- a future `availableAt` is not claimed;
- retry delay survives runner restart;
- old queued jobs become `expired`;
- active leases block other claims;
- expired leases increment retries and requeue or fail;

- orphan reconciliation runs even when no new job can be claimed;
- each attempt receives a different claim token;
- heartbeat and terminal transitions succeed only for the current token;
- run creation and terminal transitions succeed only for the current claim;
- finding and metric writes reject stale or terminal attempts;
- snapshot and reconciliation writes reject stale attempts;

- only one SQLite runner claims work under concurrent attempts;

- heartbeat loss aborts the claim context and blocks later worker commits;

## Accepted Constraints

- SQLite provides the required cross-process atomic claim guarantee.
- Flotiq cannot provide that guarantee with its current API and requires one
  designated runner process.
- Leases and fencing prevent stale storage transitions, but no storage queue can
  guarantee exactly-once external provider side effects after a network
  partition. Existing deduplication and reconciliation remain responsible for
  making retries safe.
- Reasoning effort is optional per role. ReviewPhin never invents or silently
  downgrades an effort value.
