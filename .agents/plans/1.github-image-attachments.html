Plan 01 · GitHub images

Let ReviewPhin see the screenshot a user sends.

When someone adds an image to a GitHub comment, ReviewPhin sees the words in the comment but not the image. The model may refuse to answer or guess from missing information.

ImportanceFix first
Who noticesGitHub users
Main code areaGitHub review input
New stored dataNone
Finished result: a user can attach a normal image to a GitHub pull request comment, ask ReviewPhin about it, and get an answer based on the actual image.

What is wrong today?

1. User adds a screenshotThe comment contains text plus a GitHub attachment link.
2. ReviewPhin drops the imageThe GitHub code returns an empty image list.
3. The model gets only textIt cannot honestly describe or inspect the screenshot.

This is especially bad for private repositories. The image may need GitHub App permission to download, so handing the model the web link is not enough.

What do we want instead?

  • Find normal Markdown and HTML images in the GitHub comments used by the review.
  • Download allowed GitHub images with the GitHub App’s permission.
  • Send the downloaded image to the model when that model supports images.
  • If one image fails, keep the rest of the review running and clearly tell the model which image is missing.
  • Never download random websites just because somebody put a URL in a comment.

How do we fix it?

  1. Find image links. Read image links from the comment text. Support the same common Markdown and HTML forms that work on GitLab.
  2. Check that the link is safe. Allow only known GitHub attachment locations. Check redirects too, because a safe-looking URL can redirect somewhere unsafe.
  3. Download with GitHub access. Use the repository’s GitHub App installation access. Stop if the file is too large, empty, or not a supported image type.
  4. Use the shared image path. Give the image to the same model and logging code used for GitLab images. Do not build a second model-image system.
  5. Explain failures. If an image cannot be downloaded, add a clear note to the model input. The model must not pretend it saw that image.
Safety rule: do not turn this into a general web downloader. The code must reject unknown hosts, unsafe redirects, unsupported file types, and files over the size limit.

Work steps

  1. Test real GitHub attachment links. Record how public, private, missing, and forbidden images behave. Do not log access tokens or secret parts of URLs.
  2. Share only the common image code. Reuse link parsing, file naming, duplicate removal, size checks, and result formatting. Keep GitHub login and URL rules in GitHub code.
  3. Add the GitHub download method. Stream the file so the size limit can stop a large download early.
  4. Connect it to review and chat runs. A question about an image must work even when it does not ask for a full code review.
  5. Add tests. Cover successful images, private images, redirects, bad file types, large files, partial failure, duplicate links, and models without image support.
Technical notes for the implementing agent
  • GitHubReviewRuntime.materializeAttachments() returns empty arrays today. This is the main missing connection.
  • GitLab image attachment code shows the working discovery, download-result, naming, and partial-failure shape.
  • Keep using ReviewContext.attachments, attachmentIssues, harness blob attachments, model image checks, and orchestration/image-attachments.json.
  • No storage contract, SQLite table, Flotiq type, or model-profile field should change.

How do we know it works?

  • A model with image support can correctly describe a known image attached to a private GitHub pull request comment.
  • The same image linked twice is downloaded only once.
  • A bad image does not stop good images or the text part of the review.
  • A model without image support gets the normal clear warning instead of raw image data.
  • Logs contain no image bytes, access tokens, or secret download links.
  • GitLab image tests still pass.

Checks to run: GitHub and GitLab image tests, review runtime tests, worker tests, model-session tests, prompt tests, and npm run build.
