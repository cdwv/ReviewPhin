Plan 05 · repository rules

Let a repository give rules only to ReviewPhin.

A team may want ReviewPhin to focus on security, ignore generated files, or follow a special review style. Those rules should live with the code, but they should not automatically affect every other coding agent.

ImportanceMedium
Who noticesRepository teams
Main code areaWorkspace and prompts
New stored dataNone
Finished result: a repository can add .reviewphin/instructions.md, and ReviewPhin uses those rules for both reviews and replies on GitLab and GitHub.

What is wrong today?

  • ReviewPhin can bring common agent instruction files into the review workspace.
  • There is no file name that clearly means “these rules are only for ReviewPhin.”
  • A file under .reviewphin is not something the model tool will discover by magic. ReviewPhin must read it and add it to the model input itself.
  • Project memory is not the right place for these rules. Memory is learned and stored by ReviewPhin. Repository instructions are written by the team, reviewed like code, and tied to a commit.
Example: a team can write “Focus on authentication and permission mistakes. Ignore files under generated/.” ReviewPhin should follow that rule, while another agent working in the repository does not have to.

What do we want instead?

Main file

.reviewphin/instructions.md

One clear file for review rules that apply across the repository.

Optional extra files

.reviewphin/instructions/**/*.instructions.md

More files for teams that want to split rules by topic. Load them in a stable name order.

  • Read the files from the same commit that ReviewPhin is reviewing.
  • Use the same file rules on GitLab and GitHub.
  • Keep common repository instructions, then add ReviewPhin-only instructions.
  • Keep project memory separate and clearly named.
  • Show which instruction files were used, without printing their private contents into normal logs.
  • Limit the number and total size of instruction files.

How do we fix it?

1. Find the filesLook only in the allowed .reviewphin paths at the commit being reviewed.
2. Check and order themReject unsafe paths and bad files, apply size limits, then sort names in a stable order.
3. Give them to the modelAdd clearly labeled instruction text to review and reply inputs, and record a small file list for troubleshooting.

Order of instructions

  1. ReviewPhin’s built-in safety and review rules.
  2. Common repository instructions such as AGENTS.md and GitHub instruction files.
  3. ReviewPhin-only repository instructions.
  4. Project memory, clearly labeled as learned team rules.
  5. The user’s comment that started this review or reply.

Keep a label around each source. Do not make one giant piece of text where nobody can tell which file a rule came from.

Safe file rule: never follow a symbolic link or path that escapes the review workspace. A missing .reviewphin directory is normal and should do nothing.

Work steps

  1. Write the file rules. Decide allowed paths, order, file count, total size, text encoding, symbolic-link behavior, and what happens to a bad file.
  2. Add one shared file finder. GitLab and GitHub should use the same path-checking and ordering code.
  3. Make both providers bring the files into the workspace. Test repository checkout and archive paths at the exact reviewed commit.
  4. Add the text to model input. Use the shared review and reply path. Do not assume the model tool knows what .reviewphin means.
  5. Add a small run record and user docs. Record file path, order, byte count, and whether the file was used or skipped. Explain when to use common instructions, ReviewPhin-only instructions, or memory.
Technical notes for the implementing agent
  • Workspace rules list AGENTS.md, .github/copilot-instructions.md, and .github/instructions/**/*.instructions.md.
  • Extend shared discovery instead of copying path rules into GitLab and GitHub runtimes.
  • Pass labeled content through the shared prompt/session boundary used by both the reviewer and chatter. A file in the workspace alone is not proof that the harness will use it.
  • Path-specific frontmatter is not part of this work unless there is a reusable, tested rule for it. Do not invent a new glob language.
  • Store no instruction text in the database. Use the repository workspace and a run artifact that lists safe metadata.

How do we know it works?

  • A rule in .reviewphin/instructions.md changes review and reply behavior in GitLab and GitHub tests.
  • Common instruction files still work, and ReviewPhin-only files are added in a stable order.
  • Files come from the exact commit being reviewed.
  • Project memory stays separate and no new database data is created.
  • Too many, too large, badly encoded, duplicate, linked, or unsafe files have clear tested results.
  • Run details list the files used, but normal logs do not copy private instruction text.

Checks to run: workspace tests, GitLab and GitHub runtime tests, prompt tests, review and reply harness tests, worker tests, and npm run build.
