# GitHub Provider With App Manifest Flow

## Goal

Add a GitHub platform provider that uses the platform connection store from plan 1. A GitHub platform connection represents exactly one GitHub App registration/installation for one GitHub owner account or organization.

This plan assumes the platform connection store exists, every tenant has a required `platformConnectionId`, provider methods can receive resolved tenant plus connection context, and non-`ready` connections cannot be used for tenant registration or runtime processing.

## Required Decisions

- One GitHub platform connection equals one GitHub App registration/installation.
- GitHub connection add should require/store an owner name with `--owner`.
- The GitHub setup callback must validate that the returned GitHub owner matches the owner registered for the connection.
- GitHub credentials must be stored in `platformConnectionConfigJson`, not environment variables, because multiple GitHub org/account connections must be supported.
- Setup URLs use a setup token, not only the connection id.
- Setup tokens expire after 1 hour.
- Setup tokens are invalidated after successful setup.
- GitHub tenants can only be added when the assigned connection is `ready`.
- Webhooks stay under platform-specific routes such as `/webhooks/github`.
- GitHub setup should render a simple styled HTML page, not JSON-only.

## Connection Shape

GitHub connection config should contain fields needed to complete setup and call GitHub after setup. Exact field names can follow provider conventions, but the config should cover:

- GitHub API/base URL.
- Expected owner name from CLI registration.
- Setup token metadata or reference, if stored in the connection config.
- App id.
- Client id.
- Client secret.
- Webhook secret, if GitHub App webhooks use one.
- Private key.
- Installation id.
- Owner login/name returned by GitHub.
- Owner type if available.
- Any permissions/events metadata needed for display or validation.

Credentials and secrets belong in `platformConnectionConfigJson`, with CLI output redacted.

Connection status:

- New GitHub connections start as `setup_required`.
- Successfully completed GitHub App setup changes the connection to `ready`.
- Recreate/reset flow changes the connection back to `setup_required` until setup completes again.

## CLI Flow

Register a GitHub connection through:

```sh
reviewphin platform connection add --platform github --name <name> --owner <owner>
```

Behavior:

- Create a platform connection in `setup_required` state.
- Generate a new setup token that expires after 1 hour.
- Print the setup URL the operator must open.
- The setup URL should look like `/setup/github/<setupToken>`.

Recreate/reset behavior:

- `platform connection add` should fail if a connection with the same name already exists.
- `platform connection add --recreate` should be explicit and should not remove the connection record.
- `--recreate` clears existing GitHub setup/config data needed to force setup again, creates a fresh setup token, marks the connection as not ready, and prints the setup URL.
- Tenants attached to the connection remain in storage.
- Runtime/webhook paths should fail early while the connection is not `ready`.

Tenant registration:

- `tenant add --platform github --connection <name-or-id> ...`
- GitHub tenant config must use repository numeric id as the canonical repository identifier.
- Preserve `owner/repo` full name as useful metadata for display and operations, but do not treat it as canonical because repository names can change.
- If easy, tenant add should call GitHub to verify that the repository is visible to the configured app installation. Missing access should warn the operator rather than fail the process.

## Setup Routes

Use setup routes under the GitHub platform prefix:

- `/setup/github/<setupToken>` for setup page and callback.
- `/setup/github/<setupToken>/return` may be added if a separate callback route is cleaner.

The setup page should:

- Look up the setup token.
- Reject expired, missing, invalid, or already-used tokens.
- Render a simple styled HTML page.
- Generate the GitHub App manifest with pre-filled values where possible.
- Use the stored owner value to pre-fill or constrain setup where GitHub allows it.
- Send the setup token or a derived signed state value through the GitHub flow so the callback can identify the connection.

The callback should:

- Validate the setup token/state.
- Exchange GitHub callback data for final app credentials.
- Validate that the returned owner matches the owner stored on the platform connection.
- Save final credentials into `platformConnectionConfigJson`.
- Set connection status to `ready` on success.
- Invalidate the setup token on success.
- On error, show the setup page with an error status where possible.

Owner validation is required to reduce risk that someone who obtains the setup URL configures a different GitHub account/org than the intended one.

## Tenant Keys

GitHub tenant keys should be based on GitHub API/base URL plus repository numeric id. This mirrors the readability of GitLab keys while avoiding repository-name instability.

Repository full name may be stored and refreshed later if it no longer matches GitHub, but repository id is canonical.

## Webhooks

GitHub webhooks must stay under the platform-specific route:

```text
/webhooks/github
```

Do not replace platform-specific webhook routes with a single generic `/webhooks` dispatcher.

Webhook processing should:

- Use the GitHub provider implementation selected by the route.
- Verify the webhook using the natural GitHub App mechanism. If this is an app-level webhook secret, store it in the platform connection config.
- Resolve the relevant connection/app context from webhook data such as installation id, then resolve the tenant from repository id plus connection/base URL context as needed.
- Reject or fail early if the resolved connection is not `ready`.
- Extract the pull request/review trigger data through the provider and enqueue review jobs through the existing central app flow.

The tenant key is not secret. It is a readable identifier for operators and CLI usage.

## Provider Implementation

Add a GitHub provider implementing the platform interface from plan 1:

- `getTenantRegistrationSchema()`
- `getConnectionRegistrationSchema()`
- connection add/update hooks that produce `setup_required` or `ready` as appropriate
- tenant add hook that can receive the resolved connection
- setup handler for GitHub App manifest flow
- webhook payload parsing, authorization, trigger classification, and job creation
- review runtime support needed to read pull request data and publish comments

Use provider-owned config parsing helpers, following the pattern used by existing tenant config parsing.

## Tests

Add tests for:

- GitHub connection add creates `setup_required` connection and setup URL.
- Setup token expires after 1 hour.
- Setup token is invalidated after successful setup.
- Callback rejects owner mismatch.
- Callback stores credentials and marks connection `ready`.
- `--recreate` keeps connection and tenants, clears setup data, creates a new setup token, and marks connection not ready.
- GitHub tenant add rejects non-ready connection.
- GitHub tenant add stores repository id canonically and preserves full name metadata.
- GitHub tenant key uses GitHub API/base URL plus repository id.
- GitHub webhook verification and tenant resolution.
- Runtime/webhook paths fail early for non-ready connection.
- CLI list/describe redacts GitHub secrets.
