Plan 11 · authentication docs

Use one recommended Copilot token name everywhere.

Setup examples alternate between GH_TOKEN and COPILOT_GITHUB_TOKEN. Readers cannot tell whether both are required or whether the names serve different purposes. The docs should recommend one, explain the aliases, and show their real precedence.

ImportanceMedium
Who noticesOperators

Stored dataNo change
Finished result: every ReviewPhin setup starts with COPILOT_GITHUB_TOKEN, states that only one supported variable is needed, explains which variable wins when several exist, and separates model authentication from GitHub App platform access.

What is wrong today?

Examples choose different names

Local, Docker, Kubernetes, model-profile, and environment pages do not lead with the same variable.

Aliases look like separate secrets

Showing three assignments without an explicit rule can make an operator set all three to the same or different tokens.

Precedence is missing

If several variables are inherited from a shell or CI system, the operator cannot predict which credentials Copilot uses.

GitHub has two unrelated roles

A token used to call Copilot is easy to confuse with the generated GitHub App that reads and writes pull-request data.

What do we want instead?

1. COPILOT_GITHUB_TOKENRecommended for ReviewPhin and checked first by Copilot CLI.
2. GH_TOKENSupported compatibility alias used only when the first variable is unset.
3. GITHUB_TOKENFinal fallback, often already present in CI environments.
  • Operators are told to set exactly one variable.
  • COPILOT_GITHUB_TOKEN is the canonical ReviewPhin example.
  • The precedence order is stated once on the canonical page and summarized elsewhere.
  • The token owner’s Copilot entitlement and policy requirements remain documented.
  • GitHub App credentials are clearly described as separate platform authentication.

How do we fix it?

Documentation problem Planned change
Setup snippets use different variables. Change runtime examples to COPILOT_GITHUB_TOKEN unless a page specifically teaches aliases.
The environment reference lists names without behavior. State that one variable is enough and document the exact precedence order.
Generic “GitHub API token” wording is ambiguous. Call it a Copilot authentication token and explain that per-profile tokens select model credentials, not platform credentials.
GitHub platform setup is nearby. Add a short comparison between Copilot authentication, GitHub App credentials, and the temporary App setup link.
Examples can drift again. Add a documentation check that flags non-canonical runtime snippets while allowing deliberate CI and release examples.
Do not rename runtime support: all three variables remain accepted by Copilot CLI. This is a documentation standardization, not a breaking configuration change.

Work steps

  1. Write the canonical explanation. Explain purpose, recommendation, aliases, precedence, entitlement, and GitHub App separation on the model authentication page.
  2. Normalize setup examples. Update local, Docker, Kubernetes, README, and environment examples to lead with the canonical name.
  3. Fix misleading wording. Replace “GitHub API token” where the value actually authenticates Copilot or a custom model profile.
  4. Preserve intentional exceptions. Keep release workflow documentation about CI-provided GITHUB_TOKEN because it describes a different use.
  5. Check every path. Build docs, search for all three names, and verify that each remaining occurrence either follows the recommendation or explains an exception.
Technical notes for the implementing agent
  • The canonical explanation belongs in Model profiles and the compact reference belongs in Environment variables.
  • Normalize examples in README, Run locally, Docker, and Kubernetes.
  • Cross-link Platform connections for the GitHub App distinction.
  • The current Copilot CLI reports precedence as COPILOT_GITHUB_TOKEN, then GH_TOKEN, then GITHUB_TOKEN. Verify this again with the bundled CLI help before publishing.
  • Harness session construction passes a profile token directly when one is configured; the fallback Copilot path inherits the CLI environment.

How do we know it works?

  • A new operator sees one token assignment, not three competing assignments, in every setup walkthrough.
  • The docs say explicitly that only one supported token variable is required.
  • The documented precedence matches the bundled Copilot CLI help.
  • Readers can explain the difference between Copilot authentication and the GitHub App connection.
  • CI release documentation keeps its intentional use of the workflow-provided GITHUB_TOKEN.
  • A repository-wide search finds no unexplained conflicting example.
