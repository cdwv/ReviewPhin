Plan 03 · nearby requests

Turn several quick comments into one review run.

People often leave several comments in a short burst. ReviewPhin should wait a few seconds, gather the comments for the same merge request or pull request, and handle them together.

ImportanceHigh
Who noticesBusy teams
Wait time15 seconds by default
New stored columnsNone expected
Finished result: comments that arrive close together become one job, exact duplicate webhooks are ignored, and no comment can disappear when a worker starts the job.

What is wrong today?

One burst can start several jobs

Three comments about the same code review can create three separate model runs. That costs more and may produce answers that do not know about each other.

Duplicate webhooks look like new work

GitLab or GitHub may send the same event more than once. A duplicate must not add the same request twice or restart the waiting time.

Adding a comment can race with starting the job

One process may add a comment at the same moment another process takes the job. Without one safe database action, the new comment can be missed.

The code may move while we wait

A new commit can change the pull request or merge request before the grouped review starts. ReviewPhin needs one clear rule for which code it reviews.

What do we want instead?

1. First request opens a groupThe job waits 15 seconds before a worker can take it.
2. Nearby requests joinEach real new request resets the short wait. An exact duplicate does nothing.
3. One worker closes the groupOnce the worker takes the job, later requests start the next group.
  • Group only requests for the same tenant and the same merge request or pull request.
  • Keep different tenants and different code reviews running in parallel.
  • Put a maximum number of requests and a maximum total wait on one group.
  • Choose the strongest request in the group: full review first, then a finding follow-up or normal review, then a reply-only question.
  • Reply to every comment that needs a reply.
  • Give every triggering comment the correct queued, running, success, or failure signal when its platform supports one.

How do we fix it?

  1. Use the job’s existing ready time. Set the job’s ready time to 15 seconds after the newest real request. Do not add another “start later” field.
  2. Keep the grouped requests inside the job data. Store a small versioned list in the job’s existing JSON payload. Each item keeps its duplicate key, trigger, provider payload, code revision, and arrival time.
  3. Add one indivisible database action. The database must check for a duplicate, find or create an open group, add the request, and move the ready time as one action. Nothing else may take the job halfway through that action.
  4. Close the group when a worker takes it. A running or retried job accepts no new request. The new request opens another group.
  5. Build one plan from all requests. Work out whether the group needs a review, a full review, replies, or memory updates. Keep all reply targets.
  6. Check the code revision before work starts. Use the newest checked head before the job starts. If the code changes after loading begins, follow one tested GitLab/GitHub rule instead of guessing.
Never wait forever: a steady stream of comments must not keep pushing the job back forever. Decide the maximum group size and maximum total waiting time before writing the code.

Work steps

  1. Write the rules as tests. Choose which request types can join, their priority, the duplicate key, the size/time limits, and what happens when the code changes.
  2. Define the grouped JSON shape. Make it versioned and strictly checked. A job with one normal payload must still work as a one-request group.
  3. Add the safe database action. Implement it for SQLite and for the one-worker store used by Flotiq and simple custom storage.
  4. Use it when webhooks arrive. Return a clear result: new group, added to group, exact duplicate, or new group because the first one closed.
  5. Build and run the grouped work. Merge the request types, keep all replies, and update the status of all triggers.
  6. Test races and code changes. Prove that a comment arriving as a worker takes the job ends up in either that job or the next job—never nowhere.
Technical notes for the implementing agent
  • Reuse tenantId, codeReviewId, availableAt, triggerJson, payloadJson, dedupeKey, headSha, and the claim fields in storage-v005.
  • Do not add triggersJson or startNotBefore. Read a non-group payloadJson as one group item so stored jobs need no data rewrite.
  • Add a store operation such as admitInteractionTrigger. SQLite needs a transaction or conditional update. The single-worker store may serialize the same action.
  • The new operation may require a new storage contract version even with no SQLite column or Flotiq field change. Update contract history and custom-storage guidance if it does.
  • createOrGetInteractionJob is a find followed by an upsert and cannot provide the required race guarantee by itself.
  • Parse REVIEWPHIN_JOB_DEBOUNCE with the project’s normal configuration rules.

How do we know it works?

  • Several real requests for one code review join one group when they arrive inside the limits.
  • The same webhook delivered twice is stored once and does not extend the wait.
  • A race test proves every request lands in the running group or the next group.
  • Different tenants and code reviews can still run at the same time.
  • A group cannot wait forever or grow forever.
  • Jobs stored as a single request still run without a data migration.
  • GitLab and GitHub follow the same request-priority and code-revision rules.

Checks to run: storage contract tests, SQLite tests, Flotiq tests, one-worker store tests, queue and worker tests, webhook and trigger-status tests, npm test, and npm run build.
