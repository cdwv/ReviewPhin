Plan 09 · local CLI

Review a remote change without writing back to its platform.

The CLI can submit a review of an existing merge request or pull request, but a successful run follows the normal publication path. Operators need a mode that reads remote context, runs the model, and shows the result locally while guaranteeing zero remote mutations.

ImportanceMedium
Who noticesReviewers and operators
Main areaReview CLI and worker
Stored dataLocal job and run records
Finished result: mr review --no-publish reads an existing remote review, performs the full analysis, saves local diagnostics, and prints a readable report without creating or updating anything on GitLab or GitHub.

What is wrong today?

Local submission still publishes

The command originates locally, but the worker reconciles findings into remote discussions and summaries.

The result is hidden behind job status

The watch view reports progress and counts. It does not present the completed review as a useful local report.

“Dry run” would be misleading

The operation still reads the remote platform, calls a model that may incur cost, and writes local database and run-log records.

Pretty output needs Markdown semantics

Finding bodies contain headings, lists, code, and suggestions. Printing raw markers produces a poor terminal experience.

What do we want instead?

1. Read remotelyLoad the selected merge request or pull request and hydrate its workspace.
2. Analyze locallyRun the normal reviewer and save its checked result and diagnostics.
3. Render locallySkip all publication and print the report in the requested CLI format.
  • The default command keeps its current publication behavior.
  • --no-publish means zero remote writes, not zero remote reads.
  • Pretty output renders Markdown for a terminal.
  • Plain output is valid Markdown suitable for redirection to a file.
  • JSON output returns stable structured review data.

How do we fix it?

Area Planned behavior
Command option Add --no-publish. Do not call it a dry run because model and local persistence work still happen.
Queued execution Store the publication choice with the submitted local request so another runner process sees it.
Remote safety Skip discussion reconciliation, summaries, replies, reactions, check updates, and any project-memory write.
Local persistence Complete the interaction run with the checked review result and retain normal logs and metrics.
CLI report Build one canonical Markdown report, then render it for pretty mode or emit it directly for plain mode.
Zero remote writes is a contract: tests must fail if any GitLab or GitHub mutation endpoint is called, even for progress feedback, cleanup, memory, or an error path.

Work steps

  1. Persist the publication choice. Extend the local review request with a backward-compatible publication mode and expose --no-publish in help.
  2. Fence remote mutations. Decide publication once in the worker and bypass every mutation path while retaining remote reads and workspace hydration.
  3. Keep local results useful. Save the completed result and make the watcher load it after terminal job completion.
  4. Create one report formatter. Convert overview, readiness, findings, anchors, explanations, and suggestions into canonical Markdown.
  5. Render by output mode. Use terminal styling and width-aware wrapping for pretty mode, raw Markdown for plain mode, and structured data for JSON mode. Respect color settings and non-terminal output.
  6. Prove the safety boundary. Run successful, failed, retried, GitLab, and GitHub no-publish jobs against clients that reject every mutation call.
Technical notes for the implementing agent
  • Extend localReviewTriggerSchema with a backward-compatible publication mode. Existing stored triggers must continue to mean normal publication.
  • The CLI submission and wait path lives in runMergeRequestReviewCommand and watchReviewJob.
  • The main mutation boundary is the review worker, especially discussion reconciliation and chatter publication. Local triggers already use NoOpPlatformTriggerLifecycle.
  • Parse the stored result with reviewResultSchema. Add a dedicated report module rather than coupling terminal rendering to platform summary comments.
  • Add any Markdown parser/renderer as a direct production dependency. Do not rely on documentation-tool transitive dependencies.
  • No new storage column is expected because the trigger JSON and existing interaction result are sufficient.

How do we know it works?

  • The same remote change can be reviewed with and without publication, and the default remains unchanged.
  • A no-publish run makes remote read calls but no remote mutation calls in success, retry, or failure paths.
  • Pretty output has readable headings, lists, code blocks, severity, and file anchors at narrow and wide terminal widths.
  • Plain output can be redirected to a file and remains valid Markdown without ANSI escapes.
  • JSON output contains the checked review result and job identity without mixed progress text.
  • The run result, metrics, and diagnostic artifacts remain available locally.

Checks to run: merge-request review CLI tests, review-watch tests, worker orchestration and lifecycle tests, GitLab and GitHub runtime tests, CLI output snapshot tests, pnpm test, pnpm typecheck, and pnpm build.
