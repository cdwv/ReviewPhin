# Plan: Code Review Platform Provider

## Goal

Make the code review platform fully pluggable so the app can support more than GitLab without
hardcoding provider-specific behavior into webhook handling, tenant registration, review execution,
or setup flows.

---

## Current state

The codebase already has the core platform boundary in place:

- `src/platforms/IPlatform.ts` defines the provider contract for webhook parsing, tenant key
  derivation, request authorization, trigger classification, interaction job creation, review
  runtime creation, tenant registration validation, and optional setup routes.
- `src/platforms/gitlab/` contains the GitLab implementation behind that platform boundary.
- `src/app.ts` registers webhook routes per platform via `getPlatforms()`.
- `src/tenants/tenant-registry.ts` resolves tenants by `platform` and provider-derived tenant key,
  then delegates request authorization to the platform.
- `src/jobs/review-worker.ts` already delegates trigger classification, interaction job creation,
  and review runtime creation to the platform implementation.
- `storage-v001` is active and tenants are stored as `platform` + `platformConfigJson`.
- `tenant add` and `tenant list` already work with platform-aware tenant data.

---

## Remaining work

1. **Provider loading**
   Replace the hardcoded platform registry with dynamic loading similar to storage providers, so
   built-in shorthands and external/custom platform modules can be configured instead of being
   compiled directly into `src/platforms/platform-registry.ts`.

   **Decision:** use a configurable list of platform modules/shorthands, not a single registry
   module and not a single platform module. Built-in shorthands such as `gitlab` should keep
   working.

2. **Setup route wiring**
   `IPlatform.getSetupRoutes()` exists, but `src/app.ts` does not yet register provider-declared
   setup endpoints. Wire setup into the app so providers can expose setup and onboarding flows.

   **Decision:** use the same routing approach as webhooks: one app-level setup callback per
   platform mounted under `/setup/<platform>` plus its wildcard suffix form. If a provider needs
   multiple setup pages or sub-routes, it should route internally inside that callback based on the
   matched suffix, not by registering many app routes.

3. **Raw webhook body support**
   Extend the webhook request contract to include raw request bytes. This is required for providers
   that validate signatures using HMAC or other raw-body-based mechanisms.

   **Decision:** the webhook request contract should explicitly carry raw request bytes, not only a
   parsed body.

4. **Provider-specific webhook path support**
   Keep the wildcard webhook route behavior after `/webhooks/<platform>/...` and make it part of the
   supported provider contract, so providers that cannot derive tenant identity from the payload can
   extract a tenant identifier from the route suffix instead.

   **Decision:** keep one app-level webhook callback per platform mounted under
   `/webhooks/<platform>` plus its wildcard suffix form. The webhook request contract should
   explicitly carry the matched webhook path suffix in addition to headers/body/raw body, so
   providers can use it during tenant identification and authorization or do provider-internal
   sub-routing when needed.

5. **Provider-native interaction job data**
   Review whether `InteractionJobRecord` needs an additional provider-native payload field or a more
   generic identifier model before supporting another platform cleanly.

6. **Tenant config expectations**
   Confirm whether keeping secrets inside `platformConfigJson` remains sufficient once HMAC-based or
   OAuth-style providers are added. If not, adjust tenant storage before introducing another
   provider.

   **Decision:** treat this as an analysis gate, not an automatic redesign. Keep
   `platformConfigJson` unless a concrete provider requirement proves it insufficient.

7. **Note vs Comment entity naming**
   In GitLab comments were called notes. Leftover naming still refers to noteId/note_id instead of commentId and comment_id. Since note is not the best abstract for that, we need to replace uses of note* names outside of `src/platforms/gitlab` with comment*. That includes migrations in sqlite schema (that preserve existing data) and renames in flotiq schema (we don't need migrations in flotiq or data preservation in flotiq for this task). If we need to refer to platform-specific comment id in generic code, that should be explicitly called `platformCommentId`.

8. **Thread vs Discussion**
   Both thread and discussion refer to the same abstract as far as I know. I think we should unify them to discussion. As above, if we need to refer to platform-specific discussion id and internal discussion id at the same time, it should be with explicit field called `platformDiscussionId`. Migrations should follow the same rules as in comments.

---

## Decisions made during plan review

- Treat `thread` as generic legacy terminology and rename it to `discussion` end-to-end in generic
  code outside `src/platforms/gitlab`. That includes generic contracts, prompt/building context,
  metrics, storage-facing field names, and CLI/output naming where applicable.
- It is acceptable for `src/platforms/gitlab` to keep platform-native wording internally where that
  mirrors the GitLab API, but generic interfaces exposed outside that provider should use
  `discussion`.
- If generic code needs both internal and provider-native ids at the same time, use explicit names
  such as `platformCommentId` and `platformDiscussionId`.
- Dynamic platform loading will likely require an async initialization path similar to storage
  provider loading, because the platform list should no longer be compiled as a static singleton.
- Webhook and setup routing should follow the same pattern: one app-level callback per platform for
  each route family, with the matched suffix passed through so providers can perform their own
  internal routing if they need multiple logical endpoints.
- The webhook request contract should be considered incomplete until it carries at least:
  parsed body, raw body bytes, headers, and the matched webhook path suffix.

## Recommended implementation order

1. Add raw webhook body support and provider-declared setup route registration.
2. Make provider-specific webhook path suffixes a supported and documented part of tenant
   identification.
3. Replace the hardcoded platform registry with configurable module loading.
4. Review any remaining contract or schema cleanup needed before the GitHub provider follow-up plan
   starts implementation.

---

## Done when

- Platforms can be loaded without editing `platform-registry.ts`.
- Providers can validate raw-body webhook signatures.
- Providers can register setup routes through the app.
- Providers can resolve tenant identity either from the payload or from the wildcard webhook path
  suffix.
- The platform API is ready for the follow-up GitHub provider plan in
  `.agents/plans/2.github-provider-with-app-manifest-flow.plan.md`.
