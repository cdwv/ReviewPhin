Plan 07 · usage metrics

Store every harness session with the unit it actually uses.

The metrics report still reads session files and treats premium requests as the universal cost measure. GitHub moved most Copilot plans to artificial intelligence (AI) credits in June 2026, and future harnesses may report other units. Metrics need to become stored records with explicit units, while files become an import source and optional diagnostic artifact.

Finished result: new harness sessions save their metrics without reopening a log file, the metrics report reads only from configured storage, and a separate collection command can import old session files before those files are cleaned up.

What is wrong today?

The cost label is no longer universal

The report calls the provider cost “premium requests.” GitHub changed most Copilot plans to AI credits on 1 June 2026, while some legacy annual plans still use premium requests. A single fixed label can now be wrong for two sessions produced by the same harness.

The report depends on files

The session report walks the run-log directory and parses selected files. It does not use metrics that live reviews have already saved. Deleting old run files would therefore erase reporting even when the related runs remain in storage.

Live collection also reopens a file

A harness already returns its events in memory, but the worker waits for the session file and reads it back before saving metrics. A missing or unreadable file silently prevents the stored row from being written.

One run can contain several sessions

A single interaction can run memory, review, and reply sessions. The database permits only one metrics row for that interaction, and the worker keeps only the last selected session. This loses usage before a second harness is even introduced.

What do we want instead?

1. Measure the sessionEach harness turns its native events into common counters, session identity, and a native usage unit and amount.
2. Save before finishing the runThe worker saves every completed session while it still owns the job. A metrics write failure is reported but does not fail an otherwise successful review. Log writing remains independent.
3. Report from storageReports group usage by unit. A collection command imports historical files through the same normalization rules.
  • Store one metrics record for every harness session—memory, review, reply, maintenance, and future session types—not one for whichever session happened to run last.
  • Store the harness name, a stable session key, an explicit session type, and the associated interaction run. Keep the session type open to future harness-specific values.
  • Always include session type in machine-readable results. If the human summary remains readable, add a compact breakdown of review, memory, reply, and other session types after the required unit and model summaries.
  • Treat usage unit and amount as a pair: both are present or both are absent. Absence means “the harness did not report billable usage,” not zero usage.
  • Use open, namespaced unit names so adding a harness or billing unit does not require a database revision just to extend an enumeration.
  • Never add unlike units. Show one row per run and unit, and build summaries separately for AI credits, premium requests, currency, or any later unit.
  • Let operators filter by one connection name and an optional date range. Use --from YYYY-MM-DD and --to YYYY-MM-DD as inclusive calendar dates in Coordinated Universal Time (UTC); either end may be omitted.
  • In human-readable output, give each usage unit its own run, summary, and model tables. Put sessions with no reported billable unit in a separate “usage not reported” section.
  • Show average model cost per review after first combining all sessions for the same model, unit, and interaction run. Count one review once even when that model handled several sessions in it.
  • When the terminal supports it, add one monthly stacked-bar chart per usage unit in pretty output. Each month is one bar, with model contributions stacked inside it and a clear model legend.
  • Keep token, duration, tool, prompt, and repeated-read measures available even when billable usage is unavailable.

How do we fix it?

Problem Planned change
One stored row hides earlier sessions in the same run. Change metrics to a one-to-many session history. Give every session a stable identity so retries and imports update the same record instead of creating duplicates.
Premium requests are embedded in the data shape. Replace that field with a nullable usage unit and amount. Preserve old values by marking migrated rows as legacy GitHub premium-request usage.
The worker rereads its own output file. Make each harness emit a normalized metrics envelope from its in-memory events before it returns or throws. Pass it back to the worker and save it on success and failure paths before the run becomes terminal.
The report scans a hard-coded file layout. Make metrics sessions use configured storage and its normal storage flags. Add connection-name and inclusive UTC date filters. Aggregate operational measures across sessions, but group billable usage by unit.
Mixed-unit tables would still be hard to read. Render a complete report section for each unit. Show compact distribution and model summaries by default. Put the potentially long per-review detail behind an explicit --all-sessions flag.
Tables do not make monthly model trends obvious. In pretty terminal output, draw horizontal monthly bars with model segments stacked within each bar. Scale each unit’s chart independently, show its totals and legend, and omit the chart cleanly when terminal capabilities or width are not sufficient.
Operators cannot see which tenants or connections use the most. Add complete tenant and platform-connection tables inside every unit section. On capable terminals, pair each table with a horizontal usage chart. For sessions without reported usage, chart session counts instead.
Historical files still contain useful measurements. Add metrics collect. It scans a chosen run-log root, lets the matching harness parser normalize every known session file, and upserts only records tied to an existing interaction run. Matching records are updated from the collected source. A dry run previews the result, and collection never moves or deletes a source file.
The current report can show usage by model. Preserve the harness-provided model breakdown in storage and label every breakdown with the same unit. Do not reconstruct historical prices from token counts because rates and account rules can change.
Chart fallback: summary tables are always present. The detailed per-review table appears only with --all-sessions. Charts are an extra pretty-mode view only. Show at most the eight highest-usage models as separate segments and combine the remaining models into “other”; the model table still lists every model. Use color when available and distinguish segments with Unicode shading when color is disabled. Skip charts for redirected, plain, machine-readable, non-Unicode, or very narrow output.
Resolved data policy: save one nullable primary billable unit and amount per session. Unit keys are open strings, not a database enumeration, so custom harnesses can introduce their own namespaced units. Store GitHub’s exact nano-AI-unit amount and convert it to AI credits only for display. Historical collection updates matching records but never removes its source files. The session report may make one intentional breaking output change. Metrics writes are best-effort and cannot turn an otherwise successful review into a failed review.

Work steps

  1. Encode the resolved data policy in examples. Cover a current Copilot AI-credit session, a legacy premium-request session, a bring-your-own-key session with no Copilot cost, two session types in one run, and a future harness with its own namespaced unit. Keep unit keys open rather than defining a storage enumeration.
  2. Revise the storage contract. Add session and harness identity plus an explicit session type, remove the one-row-per-run rule, add the usage pair and model breakdown, and extend filters needed for bounded reports and collection.
  3. Migrate both built-in storage providers. Preserve every old metrics row and all operational counters. Give it a deterministic legacy session key, mark its old premium-request amount with the matching unit, and attribute that amount to an unknown model. SQLite keeps both timestamps; Flotiq keeps creation time while its managed update time advances during backfill. Re-running preparation must make no further changes.
  4. Emit metrics directly from harness runs. Normalize native usage inside the harness adapter, not in the core worker. Produce metrics even when response parsing fails, and distinguish an unavailable usage value from a reported zero.
  5. Persist every live session safely. Carry the envelope through review, memory, and reply execution. Use the existing job-ownership guard for live writes, upsert by stable session identity, and record all completed sessions before the interaction run is completed or failed. Log a metrics write failure clearly, but do not fail an otherwise successful review.
  6. Add historical collection. Discover supported session files without assuming that every harness uses the Copilot directory layout. Report imported, updated, unchanged, skipped, and invalid counts. Update a matching stored session from the collected file, never move or delete source files, and make a second collection of unchanged files a no-op.
  7. Move reporting to storage. Keep per-run rows and percentiles, but show the long per-run detail only when --all-sessions is requested. Add the harness and unit where needed, merge sessions only when units match, and keep machine-readable output unambiguous when one run has several units. Resolve an optional connection name before loading its tenants. Apply inclusive UTC date bounds to the run start time, not to the time metrics were collected.
  8. Add unit tables and monthly charts. Render each unit as an independent report section. Calculate model cost per distinct review, group review usage into UTC calendar months, and add a capability-aware stacked-bar renderer for pretty terminals. Add tenant and platform-connection totals with horizontal charts, without adding charts to plain or machine-readable output. Add a session-type breakdown only when it does not crowd out the required unit and model summaries.
  9. Document the migration and cleanup gate. Update command help, operator guidance, custom adapter requirements, and release notes. Do not begin deleting historical run logs until collection coverage and stored record counts have been checked in a real deployment.

How do we know it works?

  • A new Copilot AI-credit session creates a stored metrics record with the current native unit and never reopens a log file to do so.
  • A legacy premium-request session and a bring-your-own-key session are distinguishable; missing billable usage is not saved as zero.
  • Memory, review, reply, and maintenance sessions in one interaction run each retain their own metrics and explicit session type after success or handled failure.
  • Structured output always exposes session types and their totals. Pretty output may show a compact session-type breakdown when it fits, but omitting that optional view never removes the unit or model tables.
  • The report succeeds after the matching run-log directory has been removed, because it reads configured storage only.
  • A mixed-unit report never adds AI credits to premium requests or any future harness unit. Pretty and plain output show separate run, summary, and model tables for every unit, with unreported usage kept separate. The long run table is absent by default and appears for every unit when --all-sessions is used.
  • Filtering by an existing connection name includes only reviews belonging to tenants on that connection. An unknown name fails with a clear error.
  • --from 2026-06-01 --to 2026-06-30 includes runs that started anywhere in June UTC, excludes the surrounding boundary instants, and may be combined with the connection filter.
  • Every unit’s model table shows total usage, distinct review count, and average cost per review. Reusing a model in several sessions of one review does not inflate that count.
  • Every unit includes tenant and platform-connection tables with distinct review counts, session counts, and usage totals. Capable terminals add matching horizontal charts without mixing units.
  • Pretty output on a capable terminal shows one chronological monthly stacked-bar chart per unit, with models represented by stable segments and overflow models combined as “other.” Tables remain unchanged when charts are unavailable.
  • Collection imports known historical files, skips orphaned sessions, updates matching records, never removes source files, and changes nothing on an identical second run.
  • Upgrading old SQLite and Flotiq data preserves record counts, counters, creation times, and premium-request values while both adapters report the new storage contract. SQLite also preserves update times; Flotiq records the backfill as an update. Moving between storage providers may assign new provider-managed timestamps, which is a documented limitation.
  • Existing claim fencing, tenant deletion, and cross-adapter storage migration continue to cover every metrics record.

Checks to run: focused harness and metrics command tests, worker success/failure/lease-loss tests, SQLite and Flotiq migration tests, storage migration and tenant deletion tests, npm run typecheck, npm test, npm run build, and npm run docs:build.
