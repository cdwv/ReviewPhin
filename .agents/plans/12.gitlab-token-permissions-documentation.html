Plan 12 · GitLab setup docs

State the exact GitLab role and scopes ReviewPhin needs.

The main setup guide says the token needs enough access, while the README is more explicit about Developer membership. Operators should not have to infer the minimum role or discover missing resolve permission during a review.

ImportanceMedium
Who noticesGitLab operators
Main areaPlatform setup docs
Stored dataNo change
Finished result: GitLab setup names Developer as the minimum project role, explains why ReviewPhin needs it, gives accurate scope requirements for each supported token kind, and provides a short verification checklist before tenant registration.

What is wrong today?

“Enough access” is not actionable

An operator needs a role name they can select in GitLab, not a summary of operations the token should somehow permit.

The role rule is split across pages

The README names Developer, but the canonical platform connection page does not state that minimum directly.

Token kind affects scopes

Personal, project, and group access tokens do not always receive Git-over-HTTPS access from identical scope selections.

Failure can arrive after setup

Connection registration identifies the token owner, but it does not prove that every tenant project grants discussion resolve and repository clone access.

What do we want instead?

1. Choose token kindPersonal, project, or group token with a dedicated bot identity.
2. Grant role and scopesDeveloper or higher plus API and repository access required for that token kind.
3. Verify project accessConfirm identity, read access, clone access, and discussion resolve capability.
  • Developer is the documented minimum project role.
  • The reason is concrete: ReviewPhin resolves its review discussions.
  • The API scope and repository-read requirements are stated per token kind.
  • The docs recommend least privilege without promising unsupported narrower access.
  • README and canonical docs use the same terminology.

How do we fix it?

Token setup concern Planned documentation
Project role Require Developer or higher for the token identity on every registered project.
API operations Require api access for merge-request reads, comments, updates, and resolving discussions.
Repository clone Explain the Git-over-HTTPS read scope required by personal, project, and group tokens instead of assuming one scope behaves identically for all types.
Reusable connections Remind operators that a group or personal token must have the minimum role on every tenant project attached to that connection.
Troubleshooting Map common permission errors to missing membership, API scope, repository scope, expired token, or instance policy.
Fact-check token types separately: do not preserve the current “API is the only scope” wording unless it is verified for Git-over-HTTPS cloning with every supported GitLab token kind and supported GitLab version.

Work steps

  1. Inventory real operations. List every GitLab API mutation and repository operation ReviewPhin performs during setup, review, follow-up, retry, and cleanup.
  2. Verify GitLab requirements. Check current official role and scope documentation for personal, project, and group access tokens, including self-managed differences.
  3. Rewrite the canonical setup section. Lead with the minimum role and a token-kind table, then explain why each permission is needed.
  4. Align shorter guides. Update README and deployment walkthroughs to link to the canonical requirements rather than duplicating incomplete wording.
  5. Add troubleshooting checks. Give operators a short, non-destructive checklist for identity, project membership, API access, and clone access.
Technical notes for the implementing agent
  • The canonical target is Platform connections; align the GitLab setup section in README.
  • Review actual operations in the GitLab client, especially repository cloning, discussion creation and update, and discussion resolution.
  • The publication adapter confirms that resolving bot-owned discussions is part of normal behavior, which is why Developer is the product minimum.
  • Use official GitLab access-token scope, role and permissions, Discussions API, and token-type pages as primary sources. Record GitLab version caveats when behavior differs.
  • This plan is documentation-only. Runtime permission preflight would be a separate implementation plan if desired later.

How do we know it works?

  • The canonical page says “Developer or higher” without vague substitutes.
  • The page explains that resolving review discussions requires that role.
  • Personal, project, and group token instructions identify the correct API and repository-read scopes.
  • A reusable connection warning covers access to every attached tenant project.
  • README and platform setup no longer contradict each other.
  • Every permission claim links to current official GitLab documentation and matches the operations in code.

Checks to run: GitLab operation inventory, official-source fact pass, repository-wide permission wording search, docs links, pnpm docs:build, pnpm lint, and pnpm build.
