Plan 10 · model configuration

Show which models the current credentials can actually use.

A model profile accepts any non-empty model name. The first proof that a Copilot account cannot use that model may arrive during a paid review attempt. Operators need discovery and preflight tools that reflect the credentials in use.

ImportanceMedium
Who noticesOperators
Main areaModel profile CLI
Stored dataNo change
Finished result: an operator can list the models available to the active Copilot identity, inspect useful capabilities, check a saved profile, and receive an actionable error before an unavailable model consumes a full review attempt.

What is wrong today?

Saved does not mean usable

The CLI checks that a name exists as text, not that the account or provider recognizes it.

Availability belongs to credentials

Two accounts can receive different model catalogs. A static list in documentation would become wrong for some operators.

Provider discovery is uneven

Native Copilot exposes a catalog, while custom providers do not all implement the same model-list endpoint or metadata.

Errors arrive too late

A review can spend time hydrating the repository before session creation finally reports that the model is unavailable.

What do we want instead?

1. Select credentialsUse the fallback Copilot identity or a named saved profile.
2. Discover or checkAsk the provider when supported and show capabilities it reports.
3. Configure safelyChoose an available model or receive a clear unsupported-discovery message.
  • The catalog is live and credential-specific, not hard-coded.
  • Native Copilot model listing works in pretty, plain, and JSON output.
  • A saved profile can be checked without starting a review.
  • Saving a profile remains possible without network access.
  • Providers without discovery say so honestly and preserve the provider’s runtime error.

How do we fix it?

Need Planned change
See native Copilot models. Add model-profile available-models using the current Copilot authentication environment.
Inspect capabilities. Show model ID, display name, supported reasoning effort, vision support, and other stable advertised fields.
Check a profile. Add an optional profile name to test its review and text-generation models against the selected provider when discovery is supported.
Run with a missing model. Fail before expensive hydration when a trustworthy catalog proves the selected model unavailable, and point to the discovery command.
Use a provider without discovery. Report that availability cannot be checked; do not pretend the configured name is valid or invalid.
Do not make saving depend on a live provider: profiles may be prepared offline, migrated between environments, or configured before credentials are attached. Discovery is a preflight capability, not a universal storage rule.

Work steps

  1. Define the command contract. Set selection rules, output fields, secret handling, exit codes, and the response for unsupported discovery.
  2. Expose the Copilot catalog. Start the existing client with the same authentication and CLI settings as reviews, list models, then stop it cleanly.
  3. Add profile checks. Resolve the profile exactly as a review would and validate both configured roles when the provider has a trustworthy catalog.
  4. Preflight review startup. Check availability before repository hydration where possible, while avoiding duplicate catalog calls and rate-limit pressure.
  5. Document limitations. Explain that catalogs vary by account and time, and that custom-provider discovery may be unavailable.
Technical notes for the implementing agent
  • The installed SDK exposes CopilotClient.listModels(); current production use is visible in harness session image capability checks.
  • Add the command alongside existing model-profile CLI commands and reuse the safe output layer in CliOutput.
  • Profile resolution and provider construction live in model profile resolution. Keep stored secrets masked in all output and errors.
  • Use a small service shared by the CLI and review preflight rather than creating separate catalog logic in both paths.
  • Extend CLI help tests, CLI output tests, harness client tests, and model profile tests.
  • Track the original request in GitHub issue 28.

How do we know it works?

  • The command lists only models returned for the current Copilot credentials.
  • Pretty output is readable, plain output has no ANSI escapes, and JSON output is stable and machine-readable.
  • Review and text-generation models are checked independently.
  • An unavailable native Copilot model fails before workspace hydration and suggests the discovery command.
  • An offline profile save still succeeds, and unsupported custom-provider discovery is not reported as model absence.
  • No authentication token appears in normal output, JSON, logs, or errors.

Checks to run: model profile tests, harness client tests, CLI help and output tests, review startup tests, secret-redaction tests, pnpm test, pnpm typecheck, and pnpm build.
