Plan 08 · tenant management

Never change or retain a tenant connection silently.

A remote project has one stable tenant key. Adding that project with another connection currently keeps the old assignment but can report the new connection. ReviewPhin must detect this conflict and require an explicit choice.

ImportanceHigh
Who noticesOperators
Main areaTenant CLI
Stored dataExisting assignment changes
Finished result: repeated registration on the same connection stays idempotent. A different connection triggers a prompt or an explicit flag, and command output always shows the assignment actually stored.

What is wrong today?

The key cannot distinguish connections

The key represents the remote project, so the same project produces the same key even when another credential connection is supplied.

The existing assignment silently wins

The save path deliberately preserves the first connection. An operator asking to use a second connection is not told that the request was ignored.

The success output can be false

The command can print the newly supplied connection name even though the stored tenant still points to the old connection.

Automation has no safe choice

A script cannot currently say “move this tenant” or “treat a different connection as an error.”

What do we want instead?

1. Find the tenantCompute the project key and load any existing assignment.
2. Compare connectionsContinue normally when they match; otherwise require a decision.
3. Apply one choiceReassign completely or make no change at all.
  • One remote project remains one tenant.
  • Interactive use asks before reassignment.
  • Automation can explicitly accept or reject reassignment.
  • Rejecting the conflict performs no partial update.
  • Every output mode reports the stored connection.

How do we fix it?

Situation Planned behavior
No tenant exists. Create it with the requested connection.
The tenant already uses that connection. Perform the existing idempotent update without prompting.
A different connection is supplied interactively. Show old and new connection names and ask whether to reassign.
--reassign-connection is supplied. Move the existing tenant to the requested ready connection without prompting.
--no-reassign-connection is supplied. Make no change, report the conflict, and return a non-success exit code.
No decision flag is supplied without a terminal. Make no change and explain which explicit flag is required.
Atomic choice: declining or rejecting reassignment must preserve the connection, tenant configuration, profile assignment, timestamps, and history exactly as they were.

Work steps

  1. Detect conflicts before saving. Resolve both connection records and compare stable IDs before any tenant field is updated.
  2. Add explicit CLI choices. Reject contradictory flags, prompt only in pretty interactive mode, and provide a useful non-interactive error.
  3. Add a real reassignment operation. Validate platform and readiness, then update the assignment in one storage operation while preserving the tenant identity and history.
  4. Make output truthful. Resolve the connection name from the saved tenant instead of copying the input value into the result.
  5. Document the rule. Explain one-tenant-per-project behavior, interactive prompts, and both automation flags.
Technical notes for the implementing agent
  • The conflict begins in the tenant add command, where the result currently copies the requested connection name.
  • StorageHelpers.upsertTenant intentionally preserves an existing connection. Keep ordinary upsert semantics and add a clearly named reassignment path rather than changing every caller implicitly.
  • GitLab and GitHub keys come from GitLab platform key generation and GitHub platform key generation. Do not add connection identity to those keys because webhook lookup identifies the remote project.
  • Extend platform connection tests, tenant CLI tests, CLI help, and JSON/plain output tests.
  • No storage schema migration is expected; the current tenant assignment field is sufficient.

How do we know it works?

  • Adding a new tenant and repeating it on the same connection remain idempotent.
  • An interactive conflict names both connections and defaults to no reassignment.
  • Both explicit flags work in non-interactive pretty, plain, and JSON modes.
  • Rejecting the conflict changes no tenant field.
  • Reassignment preserves tenant ID, review history, and model profile unless another option explicitly changes the profile.
  • The displayed connection always matches a fresh read from storage.

Checks to run: tenant CLI tests, platform connection tests, webhook tenant-resolution tests, CLI help and output tests, SQLite and Flotiq adapter tests, pnpm test, pnpm typecheck, and pnpm build.
