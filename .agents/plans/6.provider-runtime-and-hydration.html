Plan 06 · platform data

Fetch all needed comments, but skip work we do not need.

ReviewPhin asks GitLab and GitHub for code-review data. Some GitHub threads can be cut off after 100 replies, while some simple requests can load much more data than they need. The code also passes platform data through loosely typed containers that are easy to misuse.

ImportanceLower
Who noticesLarge or busy projects
Main code areaGitLab/GitHub runtime
New stored dataNone
Finished result: ReviewPhin reads every reply it needs, does not download a repository for a request that can be answered without it, and keeps GitLab/GitHub data in clear, checked shapes.

What is wrong today?

Long GitHub threads can be incomplete

ReviewPhin can fetch all review threads, but each thread may contain only its first 100 comments. A later reply can be missing from the model’s context.

A small request can trigger large downloads

Before ReviewPhin knows what kind of work is needed, it may load repository files, memory, discussions, and posting data. A simple ignored event or short reply should not pay for a full review.

Platform data loses its type

Some shared code carries GitLab or GitHub context as unknown. Provider code later casts it back. A wrong cast can fail only at runtime.

The same comment rules live in several places

GitHub code works out ownership, thread markers, anchors, fallbacks, and resolved state during loading and again during posting. The answers can drift apart.

What do we want instead?

1. Load just enough to chooseFind the tenant, code review, trigger, code revision, and small amount of thread data needed to understand the request.
2. Load the full review only when neededThen fetch code, workspace, memory, all thread pages, images, and review history.
3. Post with the same meaningUse one GitHub mapper for comment ownership, anchors, fallbacks, and thread state.
  • Read every page of comments inside a GitHub review thread.
  • Keep comment order stable and avoid duplicate comments where pages meet.
  • Measure API calls before changing when data loads.
  • Skip repository and full-review loading for events that do not need it.
  • Keep provider-specific fields inside provider code, but pass them in a type-safe wrapper.
  • Use one provider function to turn raw GitHub comments and threads into ReviewPhin’s common discussion shape.

How do we fix it?

  1. Measure first. Count API calls and time for an ignored webhook, a reply-only comment, a normal review, a thread with more than 100 comments, and a retry. This tells us which work is truly wasteful.
  2. Fix long threads. Add inner comment pagination to the GitHub client. Fetch until there is no next page, keep order, and remove page-edge duplicates.
  3. Make a small first-stage record. The first stage should contain only what ReviewPhin needs to decide whether to ignore, reply, or review.
  4. Load expensive data after the decision. Download the workspace, full changes, memory, attachments, and full discussions only for work that uses them.
  5. Give platform context a real type. Shared code may carry an opaque provider value, but it should not use unchecked casts scattered through the runtime.
  6. Use one GitHub comment mapper. Loading and posting must agree on which comments belong to ReviewPhin, where they are anchored, and whether they are fallback, outdated, or resolved.
Keep this focused: do not rewrite the whole platform system just to make the diagram look cleaner. Change one boundary at a time, and require a bug, test problem, or measured cost for each change.

Work steps

  1. Add measurements and fixtures. Record the API calls for the five common flows and create a thread with more than 100 comments.
  2. Add nested GitHub pagination. Test several pages, an empty last page, repeated page-edge comments, order, and API errors.
  3. Write down the small and full data shapes. Move expensive calls out of the first stage only when tests prove they are not needed there.
  4. Type one unsafe boundary. Start with the provider context that has the most casts or has caused the clearest bugs. Do not change every interface at once.
  5. Move GitHub comment meaning into one mapper. Use it from both data loading and review posting.
  6. Compare GitLab and GitHub behavior. Test common meanings such as owned, resolved, outdated, fallback, summary, and reply while keeping real platform differences visible.
Technical notes for the implementing agent
  • GitHub outer review-thread pages are fetched, but the comments inside each GraphQL thread are limited to the first 100.
  • loadRoutingContext and hydrate are separate today. Improve that split from call-count evidence instead of creating new layers by name alone.
  • IPlatform and runtime context carry platformContext: unknown. Prefer a generic or provider-owned opaque wrapper at one clear boundary.
  • Keep central review scope, job ownership, and finding rules outside providers. Keep authentication, pagination, API shapes, and marker parsing inside providers.
  • This plan changes API loading, in-memory types, and provider mapping only. It needs no storage contract or migration.

How do we know it works?

  • A GitHub thread with more than 100 comments loads every comment once and in order.
  • An ignored event and a reply-only request avoid the expensive calls they do not need.
  • Any workspace that is created is cleaned up after success, failure, cancellation, or an early return.
  • The chosen provider-context boundary is checked by TypeScript without shared code reading GitHub- or GitLab-only fields.
  • GitHub loading and posting use the same comment ownership, anchor, fallback, and thread-state rules.
  • Before-and-after API call counts show the improvement.
  • No database or stored-data type changes.

Checks to run: GitHub client and runtime tests, GitHub posting tests, GitLab runtime and posting tests, worker and cleanup tests, provider behavior tests, npm test, and npm run build.
