# Storage-Backed Job Queue Architecture

## Goal

Replace the in-memory `JobQueue` as the source of truth with persisted
interaction jobs as the queue. Webhooks, CLI commands, and future trigger
sources should only create storage rows. Worker loops claim jobs from storage
based on state.

Queued jobs older than a configurable age are skipped by the worker so stale
restored or backed-up jobs do not unexpectedly run.

## Runner

Add `src/jobs/storage-backed-job-runner.ts`.

Responsibilities:

- Poll storage for runnable jobs.
- Claim at most one job globally.
- Process the claimed job through `ReviewWorker`.
- Sleep when no job is claimable.
- Preserve ReviewPhin's current global single-review execution semantics, even
  when the HTTP server and CLI are both running.

Supported methods:

- `start()` for server process background use.
- `stop()` for shutdown.
- `runUntilJobFinished(jobId)` for CLI watch mode.
- `runUntilIdle()` for tests and maintenance.

Default poll interval: `2000ms`, configurable with:

```ini
REVIEWPHIN_JOB_POLL_INTERVAL_MS=2000
```

Default queued-job max age: 6 hours, configurable with:

```ini
REVIEWPHIN_MAX_QUEUED_JOB_AGE_MS=21600000
```

During processing, the runner renews the claim heartbeat, for example every 30
seconds.

## Stale Queued Jobs

Add a terminal job status:

```ts
type InteractionJobStatus =
  | "queued"
  | "in_progress"
  | "completed"
  | "failed"
  | "cancelled"
  | "expired";
```

Storage-backed claiming must ignore queued jobs with:

```ts
enqueuedAt < now - REVIEWPHIN_MAX_QUEUED_JOB_AGE_MS
```

Before selecting a runnable job, the runner marks expired queued jobs as:

```ts
status: "expired"
finishedAt: now
lastError: "Queued job expired before processing after <age>ms."
```

Notes:

- Expiration applies only while claiming from storage.
- Freshly created jobs should not be skipped unless the max age is configured
  too low.
- `mr describe` should show expired jobs like other historical jobs.
- Completion, failure, retry, cancellation, and expiration clear lock fields.

## Storage Contract

Introduce storage contract `storage-v005`.

Add fields to `InteractionJobRecord`:

```ts
lockedBy: string | null;
lockedAt: string | null;
lockExpiresAt: string | null;
```

Add storage helper methods:

```ts
expireQueuedInteractionJobs(input: {
  now: string;
  queuedBefore: string;
  reason: string;
  limit?: number;
}): Promise<number>;

claimNextInteractionJob(input: {
  workerId: string;
  now: string;
  lockExpiresAt: string;
  maxJobRetries: number;
  queuedAfter: string;
}): Promise<InteractionJobRecord | null>;

renewInteractionJobClaim(input: {
  jobId: string;
  workerId: string;
  lockExpiresAt: string;
}): Promise<boolean>;
```

## Claim Behavior

Claiming must be atomic:

1. Mark stale queued jobs older than `queuedAfter` as `expired`.
2. If any job is `in_progress` with `lockExpiresAt > now`, return `null`.
3. If an `in_progress` job has an expired lock, requeue it if retry budget
   remains, otherwise mark it failed.
4. Select the oldest `queued` job where `enqueuedAt >= queuedAfter`, ordered by
   `enqueuedAt`, then `id`.
5. Mark it `in_progress`, set `startedAt`, clear `finishedAt` and `lastError`,
   set lock fields, and return it.

The heartbeat renew operation must succeed only for the lock owner.

## App Changes

In `src/config.ts`, add:

```ts
jobPollIntervalMs: number;
maxQueuedJobAgeMs: number;
```

In `src/index.ts`:

- Start the HTTP app for webhook deployments as today.
- Start `StorageBackedJobRunner` beside it.
- On startup, do not manually enqueue queued jobs into local memory; the runner
  discovers fresh queued jobs from storage.
- On shutdown, stop the runner, close the HTTP app, then close storage.

In `src/app.ts`:

- Remove the `queue` option.
- Webhook route creates or deduplicates the persisted job and returns `202`.
- No in-memory enqueue call remains.

- Storage claim behavior.

- Queued jobs older than max age become `expired`.

## Assumptions

- The HTTP server still runs in normal webhook deployments.
- ReviewPhin keeps global single-review execution semantics unless a future
  change explicitly adds safe parallelism.
