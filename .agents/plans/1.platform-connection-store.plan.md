# Platform Connection Store

## Goal

Tenants must no longer store reusable platform credentials directly in their tenant config. Add a first-class platform connection store and make every tenant reference a platform connection.

This plan is the foundation for later provider-specific flows, including GitHub App manifest setup. Implementation of later plans should be able to assume the storage, CLI, runtime, and docs contracts described here.

## Required Decisions

- Every tenant must have a `platformConnectionId`. Direct tenant-only credentials are no longer supported after this migration.
- Platform connection names are globally unique across all platforms.
- Platform connection names are immutable after creation.
- Platform connection platforms are immutable after creation.
- Names are only user-facing CLI identifiers. Durable references in storage and runtime should use connection ids.
- CLI commands that accept a connection reference should accept either connection id or globally unique connection name.
- Platform connections cannot be deleted while tenants reference them. A failed delete should list the tenants that still use the connection.
- A connection whose status is not `ready` cannot be used by `tenant add`.
- `setup_required` connections must still be visible in list/describe output.
- GitLab connections are `ready` immediately after successful add/update for now.

## Storage Contract

Add a platform connection entity/store to the storage contract:

- `id`
- `name`
- `platform`
- `status`
- `platformConnectionConfigJson`
- `createdAt`
- `updatedAt`

Initial status values:

- `setup_required`: connection exists but still needs platform-specific setup before tenants can use it.
- `ready`: connection is configured and can be used by tenants.

The platform provider decides which status a newly created or updated connection should have.

Connection ids are generated by storage helpers. Storage providers must enforce
global connection-name uniqueness during initial connection creation, and
initial creation must be atomic.

Update tenants:

- Add required `platformConnectionId`.
- Keep existing tenant `platform` field.
- Keep `platformConfigJson`, but after this change it should contain only tenant/project scoped config.

Config split examples:

- GitLab tenant config: `projectId`, `webhookSecret`.
- GitLab connection config: `baseUrl`, `apiToken`, `botUserId`, `botUsername`.

For platforms that have project-level tokens or project-level metadata, those values belong in tenant config. Values that describe how this app calls the platform API, such as API URLs, access tokens, app secrets, passwords, app ids, private keys, and bot identity, belong in platform connection config.

## Storage Providers

Update both SQLite and Flotiq to the new storage contract revision.

SQLite:

- Add the platform connection table/store.
- Add required `platformConnectionId` to tenants.
- Implement a migration for existing GitLab tenants.
- Existing tenant keys must remain stable through migration.
- Enforce global uniqueness of platform connection names.

Flotiq:

- Fold the schema updates into the existing initial Flotiq migration shape rather than adding a data migration. Rename the migration file to v002 to represent it creating storage contract v002 on new setups.
- Existing Flotiq installations and data do not need a compatibility or data migration because there are no production Flotiq-store-based deployments yet.
- Pay attention to relation field use and CTD creation order so tenants can reference platform connections.

## GitLab Migration Rules

For existing GitLab tenants in SQLite:

- Move `baseUrl`, `apiToken`, `botUserId`, and `botUsername` from tenant config into platform connection config.
- Leave `projectId` and `webhookSecret` in tenant config.
- Create separate connections for distinct credential sets, not just distinct base URLs. If two tenants use the same GitLab URL but different tokens or bot users, they should become separate connections.
- Credential sets are equal when normalized `baseUrl`, plaintext `apiToken`, and `botUserId` are equal. `botUsername` is migrated into connection config but does not participate in credential-set identity.
- Tenants with equal credential sets should reference the same migrated connection.
- Generate connection names from sluggified base URL, with numeric suffixes on conflict.
- Keep `www` as part of the generated slug. If sluggification produces an empty value, use `unknown` as the base slug and apply the same numeric conflict suffixes.
- If an existing GitLab tenant config is malformed or lacks fields needed for a ready connection, still create a dedicated migrated connection and assign it to the tenant, but give the connection status `setup_required`.

Sluggification examples:

- `https://abc.zxc.com` -> `abc-zxc`
- `https://dsa-qwe.lol.pl` -> `dsa-qwe-lol`
- `https://gitlab.com` -> `gitlab`

Conflict examples:

- First matching connection: `some-host`
- Second conflicting connection: `some-host-1`
- Third conflicting connection: `some-host-2`

This conflict handling is mostly defensive because the project is not widely used yet, but the migration should not silently merge credentials that differ.

GitLab tenant keys should remain what they are today. It is acceptable for GitLab tenant key construction to use connection data, because the key stays based on normalized GitLab base URL plus project id.

## Provider Interface

Rename the tenant registration schema method and add a separate connection schema method:

- `getTenantRegistrationSchema()`
- `getConnectionRegistrationSchema()`

Add only the lifecycle hooks needed for this plan:

- connection add hook
- connection update hook
- tenant add hook

The exact method names can follow existing interface style, but the responsibilities should be clear:

- Connection add/update hooks can normalize config, infer provider-specific fields, validate credentials, and return the resulting connection status.
- Tenant add hook should receive the resolved platform connection when available, so providers can validate tenant/project config using connection credentials.
- Avoid designing a broad hook system before there is a concrete use for it.

Connection update behavior:

- The connection may be identified by id or globally unique name. The immutable `name` and `platform` fields may be supplied for identification or validation, but cannot be changed.
- Update is patch-style for mutable provider connection fields. Omitted mutable fields keep their existing values.
- Provider validation and status inference run against the combined existing and supplied config.
- A failed update must leave the existing connection unchanged.
- Provider-specific whole-connection reset/recreate flows are separate from normal update. They may preserve connection identity while wiping setup data and changing status when a later provider flow explicitly requires it.

Connection config parsing should follow the same pattern already used for tenant platform config. For GitLab, tenant JSON is parsed by provider-owned code/helpers, so GitLab connection JSON should also have provider-owned parsing helpers.

## Runtime Resolution

Runtime platform calls that need platform credentials should receive resolved connection context instead of forcing each provider to fetch it independently.

Introduce a resolved tenant concept where useful, for example:

```ts
{
  tenant: TenantRecord,
  connection: PlatformConnectionRecord
}
```

Platform methods that validate webhooks, classify triggers, create jobs, create review runtimes, or build harness context should receive the tenant, resolved connection, payload, and request context as needed. Avoid leaving connection lookup implicit inside unrelated code paths.

Webhook/runtime paths should fail early when a tenant references a connection whose status is not `ready`.

Webhook resolution behavior:

- A webhook that cannot resolve a tenant because its tenant or connection is missing or underconfigured should receive a `404`; such records are not considered eligible during webhook resolution.
- A resolved tenant/connection platform mismatch is an invalid request.
- Internal warning or error logs may record missing, malformed, mismatched, or non-ready connection state for diagnosis without exposing sensitive configuration.

## CLI

Update tenant commands:

- `tenant add` should require `--connection <name-or-id>`.
- The referenced connection must exist.
- The connection platform must match the tenant platform.
- The connection status must be `ready`.
- `tenant add --platform gitlab` should still default to GitLab when `--platform` is omitted.
- Help and docs must explain that `--connection` accepts either a globally unique connection name or a connection id.
- When `tenant add` resolves an existing tenant, it must not change that tenant's `platformConnectionId`.
- When `tenant add` resolves an existing tenant and `--model-profile` is omitted, preserve the existing model profile.
- Keep existing-tenant upsert behavior in `tenant add`; a unified tenant update command is outside this plan.

Add platform connection commands under this namespace:

- `platform connection add`
- `platform connection update`
- `platform connection remove`
- `platform connection list`
- `platform connection describe`

Command behavior:

- `platform connection update` is patch-style. Omitted fields keep their existing values.
- `platform connection update` cannot change connection name or platform.
- `platform connection remove` accepts either id or name.
- `platform connection remove` is blocked when tenants still use the connection and should list those tenants.
- Dependent-tenant deletion protection is enforced by CLI-started flows in this plan; it is not required as a general storage-store operation yet.
- `platform connection list` and `platform connection describe` must redact secrets.
- Redaction can be implemented by not showing raw `platformConnectionConfigJson`, or by delegating pretty-printing to the platform adapter.
- List and describe output should include connection id, name, platform, status, and timestamps, plus an optional provider-owned sanitized configuration summary.

GitLab CLI behavior:

- GitLab connection add requires `--name`, `--platform gitlab` (or the applicable platform default), `--base-url`, and `--api-token`.
- Bot user id and username may be omitted when they can be inferred from the GitLab API token.
- Move the existing logic that infers bot user id and bot username from the GitLab API token from tenant registration into platform connection add/update.
- GitLab connection add/update returns `ready` after successful validation.

## Code Updates

Update all code paths that currently read reusable connection data from tenant config so they instead use the resolved platform connection.

For GitLab this includes, but is not limited to:

- API client creation.
- Bot identity checks.
- Review summary instructions that mention the bot username.
- Project memory backend construction.
- Webhook authorization and trigger classification where connection data is required.
- Tenant key construction where base URL is needed.

Tenant-level GitLab data should remain limited to project and webhook-specific config.
