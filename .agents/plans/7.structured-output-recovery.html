Plan 07 · model reliability

Ask the model to correct an invalid review before failing.

A model can return text that resembles the requested review but is not valid structured data. ReviewPhin currently fails that attempt and may repeat the whole job. It should make a small number of focused correction attempts first.

ImportanceHigh
Who noticesReview requesters
Main areaModel sessions
Stored dataAttempts in run logs
Finished result: when a model returns malformed or schema-invalid review data, ReviewPhin asks it to return a corrected object, validates the correction, records each attempt, and fails clearly only after the correction limit is exhausted.

What is wrong today?

Useful content can be rejected

One missing escape or required field makes the entire response unusable even when most of the review is present.

The retry is too broad

A normal job retry reloads context and asks for a new review. That costs more and can produce different findings instead of repairing the response already produced.

The error lacks repair context

Operators see that no object matched the expected shape, but cannot easily tell how many corrections were attempted or why the last one still failed.

Blind text cleanup is unsafe

Replacing quotes or punctuation in arbitrary model text can silently alter code examples and finding content. Validation, not guesswork, must decide whether a response is usable.

What do we want instead?

1. ValidateParse the first response and check the full review shape.
2. CorrectGive the model the failure details and request only a corrected object.
3. Accept or failUse the first valid correction or stop after the fixed limit.
  • Correction attempts happen inside the review attempt.
  • Every candidate passes the same parser and schema.
  • No heuristic quote replacement is used as a success path.
  • The correction limit is small, deterministic, and tested.
  • Logs distinguish the original response from corrections.

How do we fix it?

Problem Planned change
The first response is malformed. Keep the session open and ask for a corrected object using the parser failure.
The object parses but has the wrong fields. Summarize the validation failures without exposing secrets or excessive prompt content.
A correction is still invalid. Repeat only up to the configured internal limit, then return one clear terminal error.
A later publication retry starts. Reuse the already validated saved result; do not enter correction again.
Boundary with Plan 2: this plan ends when a valid review result exists. Plan 2 owns retries after that result has been saved and publication has begun.

Work steps

  1. Define correction behavior. Set the attempt limit, the eligible parse failures, the correction prompt, and the final error format.
  2. Keep the model session available. Allow another message after an invalid response without rebuilding the entire review context.
  3. Validate every answer identically. Send original and corrected candidates through the same extraction and schema checks.
  4. Record each attempt safely. Add attempt numbers, failure reasons, duration, and model usage to the existing run artifacts and metrics without logging secrets.
  5. Cover all exit paths. Test valid-first-response, repaired JSON, repaired schema, exhausted attempts, timeout, and a later publication retry.
Technical notes for the implementing agent
  • Start with HarnessSessionRuntime.run and parseHarnessStructuredResponse.
  • The review consumer is HarnessReviewProvider.review. Chatter and memory sessions use the same harness contract and need an explicit decision about whether correction applies to them.
  • Keep whole-job retry handling as the outer recovery boundary. A failed correction may still be retried there.
  • Add focused cases to harness session tests and review provider tests.
  • This work should not change the storage contract. Use existing per-session logs and the existing completed result field.

How do we know it works?

  • A response with one JSON escaping error is corrected without restarting the review job.
  • A schema-invalid response receives concise, useful validation guidance.
  • A valid first response causes no correction call.
  • Exhausted correction attempts produce one stable error and follow normal job retry rules.
  • No test accepts content because of blind quote or punctuation replacement.
  • A saved valid result is reused during publication recovery without another model call.

Checks to run: harness session tests, review provider tests, worker retry tests, run-log and metrics tests, pnpm test, pnpm typecheck, and pnpm build.
