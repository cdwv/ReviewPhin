# Cross-Platform Review Publication Polish

## Goal

Polish GitLab and GitHub review publication after the GitHub happy path is
working. These items are intentionally outside the current GitHub merge request
unless production testing shows that one blocks normal review publication.

## Scheduling And Revision Consistency

- Enforce and test the system invariant that only one job runs for a given
  tenant and pull request or merge request at a time.
- Keep parallel execution across different tenants and different code reviews.
- Make the claim durable across multiple ReviewPhin processes instead of
  relying only on the in-memory queue.
- Define an atomic storage operation for create-or-get job deduplication.
  Verify equivalent uniqueness behavior in SQLite, Flotiq, and custom adapter
  guidance.
- Make same-head Check Run provisioning idempotent under concurrent duplicate
  pull request webhook deliveries.
- Define one provider-neutral policy for a head SHA changing while a review is
  running.
- Decide whether a stale run should publish against its reviewed revision,
  restart, merge into a queued newer job, or be abandoned.
- Apply the same policy to GitLab and GitHub.

## Finding Anchor Changes

- GitLab and GitHub can edit an existing comment body but cannot move the
  existing discussion to another file or line.
- Detect when a finding matched to an existing discussion changes anchor.
- Publish a replacement discussion at the new anchor.
- Mark the old finding completed in storage.
- Resolve the old discussion when the platform supports it; otherwise leave a
  clear provider-specific completion marker or follow-up note.
- Never persist a new anchor against an unchanged platform comment.

## Fallback Discussions

- Add explicit capabilities to platform discussions, including update, reply,
  resolve, suggestion, and anchor replacement.
- Define an explicit user-facing way to target a GitHub fallback finding from
  the flat pull-request issue-comment stream. ReviewPhin bot continuations are
  linked today, but GitHub does not expose native issue-comment thread
  identity for associating an arbitrary human comment with one fallback
  finding.
- Consolidate fallback continuation marker parsing and ownership checks with
  the provider-owned discussion mapper.
- When a fallback discussion cannot be replied to or resolved, complete the
  finding in storage without failing the review run.
- Design a shared GitLab/GitHub visual marker for findings that ReviewPhin
  considers fixed but the platform cannot resolve natively.
- Filter ReviewPhin-owned summaries and fallback findings from generic review
  comment context when they are already represented as summaries or
  discussions, so prompts do not receive duplicate provider-owned content.

## Suggestions

- Support deletion suggestions by allowing an empty replacement.
- Preserve meaningful leading and trailing replacement whitespace.
- Add broader tests for multi-line replacement and platform formatting.

## GitHub File-Level Comments

- Verify whether `subject_type: file` works inside GitHub's batch create-review
  payload with installation authentication.
- If supported reliably, use file-level review comments when the file is known
  but no line can be anchored.
- Otherwise retain marked issue comments as the fallback.

## Recovery

- Cover ambiguous create and submit outcomes.
- Cover partial fallback publication and retry recovery.
- Recover when a retried model run produces a different finding set after a
  submitted marked review already exists for the same interaction job.
- Persist or reuse the generated review result before publication retries so
  idempotency does not depend on deterministic model output.
- Clean up only confirmed provider-owned drafts or pending reviews.
- Never delete submitted reviews.

## Hydration And Pagination

- Paginate comments within GitHub GraphQL review threads instead of limiting
  each thread to the first 100 comments.
- Separate lightweight routing data from full review hydration so routing does
  not download the repository archive, project memory, and all publication
  inputs before a review is known to be needed.
- Consolidate duplicated GitHub comment, anchor, ownership, and discussion
  normalization in one provider-owned mapper used by hydration and
  publication.
- Add contract tests that compare GitLab and GitHub normalized discussion
  state, including resolved and outdated threads.

## Platform API Cleanup

- Split the broad `IPlatform` surface into focused registration/setup,
  authenticated webhook, trigger lifecycle, review runtime, publication, and
  project-memory contracts.
- Replace comment-specific runtime methods with provider-owned trigger
  validation so manual/system triggers do not implement no-op comment APIs.
- Replace opaque `platformContext: unknown` plus provider casts with a typed
  provider-owned context boundary or normalized routing/hydration records.
- Model unsupported connection update behavior explicitly so CLI help does not
  advertise GitHub update options that always fail.
- Construct one run-scoped review runtime after run artifacts exist, while
  keeping any preflight routing client deliberately separate and minimal.
- Keep review policy in the central review/reconciliation layer, repository
  instruction and model-session behavior in the harness, and API rendering,
  authentication, hydration, and publication mechanics in platform providers.

## Setup And Operability

- Add deployment coverage for `PUBLIC_URL` path prefixes through a real
  prefix-stripping reverse proxy, including setup callback, installation
  callback, webhook delivery, and badge download.
- Move open-pull-request Check Run backfill into an explicit idempotent
  registration operation with progress reporting, bounded concurrency, and
  retry diagnostics for repositories with many open pull requests.
- Replace setup-token scans across every GitHub connection with a dedicated
  indexed setup-session lookup, storing only a token hash where practical.
- Validate and report the installation's effective permissions needed for
  checks, review publication, and project memory before marking a connection
  fully operational.
- Surface retained previous-app cleanup metadata in connection status or
  describe output without exposing credentials.
