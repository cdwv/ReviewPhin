## Plan: Official Website And Docs Site

Build ReviewPhin's public documentation as a modern official website plus full documentation hub. Use Astro with Starlight so the homepage can feel like a polished product site while the docs keep strong navigation, search, accessibility, and static output for both GitHub Pages and the built container image. Keep one documentation source, generate separate static outputs for GitHub Pages and the container-served image path, and create a reusable docs authoring skill so future documentation work follows the same structure, language, visual standards, and fact-checking workflow.

**Steps**

1. Choose the docs site source boundary.
   - Create a dedicated source directory such as `docs-site/` for Astro, Starlight, components, content, and site-only assets.
   - Keep generated output out of source-controlled authoring paths.
   - Use `docs/` Markdown files as migration input, not as the required final navigation model.
   - Treat `public/` as runtime static output for the server image, not as the main authoring location.

2. Add the Astro and Starlight framework foundation.
   - Add Astro, Starlight, and supporting packages through the existing `pnpm` toolchain.
   - Configure Starlight docs routes under `/docs/`.
   - Configure the official website homepage as `index.html` served under `/`.
   - Keep every documentation page and generated docs asset under `/docs/*`.
   - Ensure generated website paths do not conflict with existing API and setup routes such as `/healthz`, `/setup/*`, `/webhooks/*`, and `/github/setup/*`.
   - Enable Starlight local search from the first docs release.
   - Configure two base/output modes:
     - GitHub Pages output with a repository-aware base path.
     - Container output that serves the official homepage at `/` and docs under `/docs/*`.
   - Keep the build compatible with Windows local development and Linux CI/Docker builds.

3. Create the ReviewPhin documentation skill before broad content work.
   - Add a new skill under `.agents/skills/`, recommended `.agents/skills/reviewphin-docs/SKILL.md`.
   - The skill must be used whenever an agent creates, rewrites, fact-checks, or updates ReviewPhin docs pages or official-site copy.
   - The skill must contain the canonical documentation rules in one place, including structure, plain language, visual guide requirements, platform/provider equality, GitLab-first ordering, fact-checking, and model collaboration rules.
   - The skill should define a topic-oriented workflow: gather facts, identify reader goal, choose target page, write or update content, add visuals where useful, verify links and commands, and record source assumptions.
   - The skill should make information-oriented pages and sections eligible for new skill usage during plan execution, especially architecture, CLI, provider, storage, and deployment pages.
   - Optional split: add a second focused skill such as `.agents/skills/reviewphin-docs-factcheck/SKILL.md` only if the main docs skill becomes too large. Default recommendation: start with one strong `reviewphin-docs` skill.

4. Define the multi-model documentation workflow.
   - Use GPT-5.5 for fact gathering, information extraction, command/reference inventory, architecture tracing, and fact-checking new and existing content.
   - Use Opus 4.8 for UI/UX direction, homepage composition, page-level copywriting, narrative clarity, and visual/interactive guide design.
   - For each substantial docs page, use this order:
     - GPT-5.5 gathers facts from code, tests, README, existing docs, and runtime behavior.
     - GPT-5.5 produces a concise fact brief with source paths and open questions.
     - Opus 4.8 turns the fact brief into plain-language page copy and visual structure.
     - GPT-5.5 fact-checks the resulting page against repository facts and flags unsupported claims.
     - Opus 4.8 performs final copy polish without adding new factual claims.
   - For visual-heavy pages, use the local `frontend-design` skill alongside the new `reviewphin-docs` skill.
   - For storage-contract pages, use the existing `schema-verification-skill` alongside the new `reviewphin-docs` skill.

5. Keep writing and visual style rules in one canonical place.
   - Put all documentation writing and visual rules in the new `reviewphin-docs` skill and reference that skill from the plan instead of duplicating style rules across pages.
   - The canonical rules must include:
     - Write in plain, direct language for operators and maintainers.
     - Keep the docs pleasant and friendly without becoming chatty or vague.
     - Avoid corporate language, empty hype, and common SaaS marketing tropes.
     - Explain what a reader can do before explaining how the code is built.
     - Use short headings and task-oriented page sections.
     - Prefer "merge request or pull request" when content applies to both.
     - List GitLab first in navigation, examples, and comparison tables, while giving GitHub and custom providers equal structure and depth.
     - Treat storage adapters equally: SQLite first where it is the simplest/default path, then Flotiq, then custom adapters.
     - Avoid overusing internal type names, class names, and implementation references in beginner and task pages.
     - Keep exact command names, environment variables, file paths, and API terms accurate in reference pages.
     - Use visual and interactive guides where they reduce confusion: terminal walkthroughs, setup timelines, architecture diagrams, comparison tables, checklists, and copyable command blocks.
     - Keep visuals static-site friendly and accessible, with reduced-motion support for animations.
     - Do not let marketing language replace operational clarity.
     - Use small dolphin or code-review jokes only when they fit the page naturally, and skip them when they would distract from setup, reference, or troubleshooting content.
   - Any additional page-specific voice or design choices must point back to this canonical skill section.

6. Define the official homepage experience.
   - Build `/` as the official homepage for ReviewPhin, not a generic docs landing page.
   - Include a proper visual hero section with a clear ReviewPhin identity, short plain-language value proposition, and primary calls to action for quickstart and docs.
   - Add an animated terminal in the hero that cycles through:
     - platform registration,
     - model profile definition,
     - adding a project,
     - a review trigger/result moment.
   - Keep terminal commands illustrative and simple, avoiding long internal type names or implementation-heavy references.
   - Include concise feature sections for supported platforms, model choices, storage choices, and how ReviewPhin fits into MR/PR workflows.

7. Create the documentation information architecture.
   - Structure the docs so GitLab appears first because it is the current primary/default platform, while GitHub and future providers receive equal conceptual treatment.
   - Avoid making the docs feel GitLab-only; use neutral terms such as "merge request or pull request" where both apply.
   - Proposed top-level documentation structure:
     - `docs/index.md` — documentation landing page and reader paths.
     - `docs/quickstart/index.md` — fastest local or container setup path.
     - `docs/quickstart/gitlab.md` — GitLab-first walkthrough.
     - `docs/quickstart/github.md` — GitHub walkthrough with setup samples linked where relevant.
     - `docs/using-reviewphin/index.md` — how to use ReviewPhin in code review.
     - `docs/using-reviewphin/merge-requests.md` — GitLab merge request workflow.
     - `docs/using-reviewphin/pull-requests.md` — GitHub pull request workflow.
     - `docs/using-reviewphin/comments-and-triggers.md` — manual comments, commands, and reruns.
     - `docs/cli/index.md` — CLI overview and common tasks.
     - `docs/cli/reference.md` — complete command reference migrated from existing CLI docs.
     - `docs/configuration/index.md` — environment variables and runtime configuration.
     - `docs/configuration/model-profiles.md` — model provider setup.
     - `docs/configuration/storage.md` — storage setup and operations.
     - `docs/architecture/index.md` — plain-language system overview.
     - `docs/architecture/review-flow.md` — webhook to job to review publication flow.
     - `docs/architecture/providers.md` — provider abstraction overview.
     - `docs/providers/platforms/index.md` — platform providers overview.
     - `docs/providers/platforms/gitlab.md` — GitLab platform provider details.
     - `docs/providers/platforms/github.md` — GitHub platform provider details.
     - `docs/providers/platforms/custom.md` — custom platform provider guidance.
     - `docs/providers/storage/index.md` — storage provider overview.
     - `docs/providers/storage/sqlite.md` — SQLite storage details.
     - `docs/providers/storage/flotiq.md` — Flotiq storage details.
     - `docs/providers/storage/custom.md` — custom storage adapter guidance.
     - `docs/deployment/index.md` — Docker, public URL, data directories, and production notes.
     - `docs/deployment/github-pages-and-image-docs.md` — how the docs are published in both targets.
     - `docs/contributing/docs.md` — writing style, visual style, model workflow, and docs contribution rules.

8. Migrate and rewrite existing content into the new structure.
   - Use current Markdown pages as content bases:
     - `docs/CLI.md` for CLI overview/reference.
     - `docs/model-providers.md` for model profile configuration.
     - `docs/storage-providers.md` for storage provider docs.
     - `docs/code-review-platform-providers.md` for platform provider docs.
   - Use the new `reviewphin-docs` skill before rewriting each content area.
   - Use GPT-5.5 to extract facts from the existing page and nearby source code before Opus 4.8 rewrites the content.
   - Break long reference pages into task-oriented pages plus full reference pages.
   - Keep detailed provider extension guidance, but introduce it with simpler conceptual explanations first.
   - Make GitLab the first platform in navigation and examples while keeping provider page templates parallel across GitLab, GitHub, and custom providers.

9. Add visual and interactive guide components.
   - Add Starlight/Astro components for reusable visual guides:
     - animated terminal walkthrough,
     - platform/model/storage setup cards,
     - MR/PR review lifecycle diagram,
     - provider capability comparison table,
     - environment variable checklist,
     - copyable command blocks,
     - step-by-step setup timeline,
     - callouts for required secrets and public URL requirements.
   - Use diagrams for architecture and review flow instead of only prose.
   - Prefer short labels and plain explanations over internal class or type names.
   - Keep interactive content static-site friendly: no server dependency, no runtime API dependency, and graceful no-JavaScript fallback where practical.

10. Wire docs builds into the repository lifecycle. Add scripts for local docs development, docs production build, GitHub Pages build, and container docs build. Ensure container docs build writes the official homepage to `public/index.html` and documentation output under `public/docs/` before Docker copies `public/` into the runtime image. Ensure GitHub Pages build writes to a separate artifact directory such as `dist-site/` or `dist-docs/`. Add generated docs output directories to `.gitignore` unless the project intentionally chooses to commit built docs. Keep the existing application build intact and make docs build failures visible in CI.

11. Update Docker and server integration only where needed. Reuse the existing Fastify static serving pattern from `src/app.ts` whenever possible. Confirm that the built container homepage is reachable at `/` and docs are reachable under `/docs/*`. Confirm docs and website paths do not interfere with webhook, setup, health, or GitHub setup sample routes. Preserve existing `public/github/setup/` assets and setup sample behavior.

12. Add GitHub Pages publishing. Add a GitHub Actions workflow that installs dependencies with pnpm, builds the docs site for Pages, uploads the static artifact, and deploys it with GitHub Pages. Use the correct base path for the repository Pages URL. Keep the workflow independent from Docker publishing except for sharing the same docs source.

13. Add docs quality checks. Add a docs build check to CI. Verify Starlight local search is enabled and indexes the generated docs. Add link checking if practical, especially for links migrated from README and existing Markdown files. Add formatting/linting coverage for docs source if the chosen Astro/Starlight setup supports it cleanly. Add a lightweight visual smoke check for the homepage at desktop and mobile sizes before release. Add a fact-checking pass for each substantial migrated page using GPT-5.5. Add a copy/design pass for the homepage and key guide pages using Opus 4.8.

14. Update existing entry points and references. Update README and DockerHub README links so they point readers to the new official docs site structure. Keep README focused on short project introduction and quickest path to trying ReviewPhin. Keep detailed setup, providers, CLI, and architecture content in the docs site. Add a contributor note explaining where documentation source lives, how to preview it, which docs skill to use, and how model responsibilities are split.

**Decisions**

- Use Astro plus Starlight instead of VitePress because the site must act as the official ReviewPhin website, not only a developer documentation portal.
- Use one documentation source and two generated static outputs: one for GitHub Pages, one for the container-served image path.
- Serve the official website homepage at `/`; serve every documentation page and docs asset under `/docs/*`.
- Treat route safety as a requirement: docs and website paths must not conflict with existing API, setup, webhook, health, or GitHub setup sample routes.
- Enable Starlight local search immediately.
- Keep platform and adapter docs structurally equal, but order GitLab first in navigation and examples.
- Create a reusable `reviewphin-docs` skill so future documentation updates follow the same rules and workflow.
- Store writing and visual rules in the docs skill as the single canonical location; the plan and pages should refer to that skill instead of redefining the rules in multiple places.
- Make the docs voice friendly, plain, and pleasant; avoid corporate phrasing and forced jokes.
- Use GPT-5.5 for facts and fact-checking; use Opus 4.8 for UI/UX, homepage experience, copywriting, and final page polish.
- Use existing Markdown as source material, but allow the new structure to be reader-oriented rather than file-history-oriented.
- Prefer visual and interactive guides wherever they reduce setup confusion or explain system flow better than prose.
- If docs versioning is enabled, each docs version must match the Docker image tag produced by the same build: release builds use the version tag, merge request builds use `mr-xxx`, and pull request builds use `pr-xxx`.

**Scope Boundaries**

- Included: official homepage, docs framework, docs structure, migrated content plan, visual guide components, new docs skill, multi-model docs workflow, GitHub Pages publication, container-served docs output, README/DockerHub link updates, verification plan.
- Excluded: changing ReviewPhin runtime behavior beyond static docs serving, changing platform/provider APIs, rewriting all product copy outside the docs site, implementing new app features for setup flows.

**Further Considerations**

1. Decide whether to enable versioned docs in the first docs release or defer them until users need older release docs. If enabled, docs versions must be generated from the same value used for the Docker image tag: version tag for release builds, `mr-xxx` for merge request builds, and `pr-xxx` for pull request builds.
2. Decide whether preview docs for merge requests and pull requests should be publicly indexed or published with noindex metadata.
3. Decide whether the official homepage at `/` should be included in container deployments immediately or guarded behind a configuration flag during the first rollout.
