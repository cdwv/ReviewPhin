Plan 02 · review comments

Finish posting the same review after something goes wrong.

ReviewPhin can create a good review result and then fail while posting it to GitLab or GitHub. A retry must continue that exact review. It must not ask the model for a different review or post the same comment twice.

ImportanceHigh
Who noticesEveryone reading a review
Main code areaReview posting
New stored columnsNone expected
Finished result: after a timeout, crash, or partly failed post, ReviewPhin checks what is already on the merge request or pull request and posts only the missing pieces of the same saved review.

What is wrong today?

Posting can stop halfway

A review may contain five findings. If the first two comments are posted and the third request fails, the merge request or pull request is left with only part of the review.

A timeout does not tell us what happened

GitLab or GitHub may create a comment and then the network times out before ReviewPhin gets the answer. ReviewPhin does not know whether it should create that comment again.

A retry can produce different findings

If the model runs again, it may return different words or a different list of findings. The retry can no longer safely continue the first posting attempt.

Some review comments lose useful behavior

Deletion suggestions are rejected because the replacement is empty. Important spaces can be trimmed. Comments that cannot attach to a line fall back to a flat comment that is harder to reply to or mark resolved.

The summary comment also needs one clear link that teaches users how to ask for another review, a full review, or a follow-up. We do not need a separate help command for that.

What do we want instead?

1. Save the resultStore the exact model result before posting any comment.
2. Post what is missingUse ReviewPhin markers and saved links to find comments that exist.
3. Retry safelyReuse the saved result and continue. Never run the model again for the same posting attempt.
  • One review run produces one saved result.
  • Every retry uses that saved result.
  • ReviewPhin can tell its own comments from human comments.
  • A retry adds missing comments but does not duplicate comments that are already there.
  • Suggestions keep the exact replacement text, including an empty replacement for deleting code.
  • The summary shows a friendly link to the interaction guide.

How do we fix each problem?

Problem Fix
The post stops halfway. Save the full review first. On retry, list ReviewPhin’s comments and post only findings or the summary that are missing.
The API call times out after it may have worked. Look for the marked comment or pending review before trying the create call again.
The model could give a different answer on retry. If a saved result exists, skip the model completely and reuse the saved JSON.
A finding moved to another line. Post a new comment at the new line, mark the old finding finished, and resolve the old thread when the platform allows it.
A comment cannot be attached to a line or resolved. Use a clearly marked fallback comment. Finish the stored finding even when the platform has no matching button or thread feature.
A suggestion deletes code or needs exact spaces. Allow an empty replacement and stop trimming meaningful replacement text.
Users do not know what they can ask. Add “Ways to ask ReviewPhin for another pass, a full review, or a follow-up” to the summary, linking to the public guide.
Ownership rule: ReviewPhin may clean up only comments or draft reviews that it can prove it owns. It must never delete a human comment or a submitted review.

Work steps

  1. List every failure point. Test failure before the first comment, after some comments, after a comment may have been created, while submitting a GitHub review, and while updating the summary.
  2. Save the model result before posting. Write the exact checked result while the job still owns the run.
  3. Make retries inspect the platform first. Find ReviewPhin’s marked comments, draft reviews, submitted reviews, and summary. Continue from what exists.
  4. Make comment features honest. Give the posting code a simple way to say whether a platform can update, reply, resolve, show a suggestion, or move an anchor. Use a fallback when it cannot.
  5. Fix suggestion text. Test deletion, leading spaces, trailing spaces, and several replacement lines on GitLab and GitHub.
  6. Add the guide link. Put the same friendly link in every review summary.
Technical notes for the implementing agent
  • Use InteractionRunRecord.resultJson for the saved result. Do not add a second result column.
  • The store may need a claim-checked method such as saveInteractionRunResultForClaim. This can require a new storage contract version even though SQLite and Flotiq need no new field.
  • Use review findings, discussion mappings, job/run IDs, and platform ownership markers to find work that was posted.
  • reviewSuggestionSchema must allow an empty replacement. Review renderers must preserve the replacement text.
  • The guide target is Comments and triggers.
  • Keep the tests that cover a finding moving to a new anchor. That behavior is part of the goal.

How do we know it works?

  • After the model result is saved, a retry makes no new model call.
  • Every tested failure point ends with one complete set of findings and one summary.
  • A timeout after a successful create does not cause a duplicate comment.
  • ReviewPhin never deletes a human comment or a submitted review.
  • Deletion suggestions and suggestions with important spaces render correctly.
  • A finding that cannot use a native thread still gets a clear fallback and does not fail the whole review.
  • Every summary contains the working, friendly guide link. There is no help command or “help shown” database state.

Checks to run: posting adapter tests, discussion tests, GitHub requested-action tests, job ownership tests, summary tests, storage tests if the contract changes, npm test, and npm run build.
